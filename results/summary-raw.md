📌 **Основные темы:**

- **Агенты переоценены: процедурная память для порядка в операционном хаосе** — доклад Mikhail Shvedkov про оверхед токенов при использовании агентов и способы его снижения через вынос детерминированной логики.
- **Data Sculptor — демонстрация возможностей** — презентация Andrey Mospanov самописного Python-приложения для пакетной обработки ~1000 исторических Excel-файлов, созданного с помощью Codex GPT.
- **Инструменты оркестрации и workflow** — обсуждение фреймворков для построения детерминированных пайплайнов вокруг агентов.

✅ **Принятые решения и рекомендации:**

- **Разделение творческой и детерминированной частей.** Творческие задачи (генерация текстов, интерпретация) оставлять LLM; последовательные и повторяющиеся операции выносить во внешние скрипты/тулы.
- **Промежуточный DSL как компромисс.** Вместо генерации HTML напрямую модель готовит псевдо-разметку (DSL), а отдельный скрипт собирает из неё финальный артефакт. Это снижает расход токенов примерно вдвое при сохранении качества.
- **Ограничение контекста скиллов/тулов.** Не давать агенту доступ к инструментам, не нужным для текущей задачи — каждый лишний MCP/тул крадёт место в контекстном окне и ухудшает качество выбора.
- **Skill для повторяющихся задач.** Любую операцию, выполняемую чаще одного раза, имеет смысл оформить как skill.
- **Измерение токенов через Cursor.** Встроенная аналитика Cursor показывает разбивку по группам (system prompt, tools, MCP, conversation, generation) — использовать для аудита оверхеда.
- **Безопасность данных.** Не отправлять внутренние транскрипты, клиентские данные и чувствительную информацию во внешние LLM-сервисы (Cursor, Codex, Minimax и пр.) без предварительной анонимизации.
- **Data Sculptor как разовый, но переиспользуемый инструмент.** Подходит под дальнейшую автоматизацию через CLI-обёртку для использования агентом как тула.
