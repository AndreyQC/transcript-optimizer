# Conventions — правила ведения `-=tasks=-`

Регламент работы с задачами и документами фаз в каталоге `-=tasks=-`.
Цель — сохранять читаемую историю решений, разделять «план» и «результат»,
а также делать документы самодостаточными.

Данный файл адаптирован под текущую структуру проекта transcript-optimizer:
задачи группируются по датам, а не по фазам; фазы и чекпоинты живут в
собственных каталогах.

---

## 1. Структура каталога

```
-=tasks=-
├── TASK_CONVENTIONS.md              # этот файл — регламент ведения
├── 2026-07-09/                      # задачи за конкретную дату
│   ├── 20260709_001_idea.md
│   └── 20260709_002_plan.md
├── 2026-07-13/
│   ├── 20260713_001_dark_and_day.md
│   ├── 20260713_002_stage4_llm_summary.md
│   └── 20260713_003_stage4_llm_summary_result.md
├── 2026-07-17/
│   ├── 20260717_001_markdown_editor.md
│   └── 20260717_001_markdown_editor_result.md
-=PHASES=-
└── Phase_06.md                       # итоговый свод по фазе
-=CHECKPOINTS=-
└── 20260719_001_checkpoint.md       # сессионный снапшот
```

- **Задачи** — по датам: директория `YYYY-MM-DD/`, файлы `YYYYMMDD_NNN_<topic>.md`.
- **Фазы** — в `-=PHASES=-/Phase_NN.md` (двузначный номер: `Phase_01`, `Phase_02`, ...).
- **Чекпоинты** — в `-=CHECKPOINTS=-/YYYYMMDD_NNN_checkpoint.md`.

---

## 2. Жизненный цикл документа

### 2.1. Стандартный цикл: plan → result

Для задач, требующих реализации, предпочтительна пара документов:

- `YYYYMMDD_NNN_<topic>_plan.md` — что делаем, почему, какие решения приняты.
- `YYYYMMDD_NNN_<topic>_result.md` — что сделано, проверки, известные ограничения,
  ссылки на коммиты.

Если план и результат умещаются в один документ без потери читаемости —
допустимо использовать `YYYYMMDD_NNN_<topic>.md` без суффиксов.

### 2.2. Draft → final (для сложных архитектурных решений)

Для спорных или открытых архитектурных вопросов можно использовать пару
`_draft.md` → `_final.md`:

- `draft` содержит полную проработку и плейсхолдеры `USER_INPUT` с рекомендацией ИИ.
- `final` создаётся после закрытия всех `USER_INPUT`.
- `final` не редактируется молча; существенные изменения — через новый `draft`
  (например `..._v2_draft.md`).
- После создания `final` соответствующий `draft` **не удаляется** — остаётся
  для истории обсуждения.

---

## 3. Именование документов

| Тип | Шаблон | Пример |
|---|---|---|
| Идея/концепция | `YYYYMMDD_NNN_idea.md` | `20260709_001_idea.md` |
| План | `YYYYMMDD_NNN_<topic>_plan.md` | `20260709_002_plan.md` |
| Результат | `YYYYMMDD_NNN_<topic>_result.md` | `20260713_003_stage4_llm_summary_result.md` |
| Чекпоинт | `YYYYMMDD_NNN_checkpoint.md` | `20260718_001_checkpoint.md` |
| Анализ | `YYYYMMDD_NNN_analysis_<topic>.md` | — |
| Draft | `YYYYMMDD_NNN_<topic>_draft.md` | — |
| Final | `YYYYMMDD_NNN_<topic>_final.md` | — |

- `NNN` — трёхзначный порядковый номер внутри дня (001, 002, ...).
- Один `NNN` может иметь несколько связанных файлов (например, план и результат).
- Регистр: `snake_case` для имён файлов; технические идентификаторы — английские.

---

## 4. Содержание документов

Каждый документ должен быть **самодостаточным** — читаемым без контекста чата:

- **Заголовок H1** с темой и пометкой (plan / result / idea и т.д.).
- Вверху — блок «Контекст» со ссылками на связанные документы (относительные пути):
  ```markdown
  > Контекст:
  > - `-=tasks=-/YYYY-MM-DD/YYYYMMDD_NNN_<topic>.md`
  > - `LESSONS_LEARNED.md`
  ```
- Для планов — явные разделы: цели, решения, проверки, риски.
- Для результатов — что сделано, проверки, известные ограничения.
- Дата в шапке (`Дата: YYYY-MM-DD`).
- Русский язык для основного текста; технические термины/идентификаторы — английские.

---

## 5. Связь с фазами и чекпоинтами

- **Phase_NN.md** — краткий свод по фазе, ссылается на ключевые task-файлы и
  коммиты. Не дублирует содержимое планов/результатов, а даёт высокоуровневую
  картину.
- **YYYYMMDD_NNN_checkpoint.md** — сессионный снапшот «где мы сейчас» для новой
  сессии агента. Не дублирует phase-файл; см. `-=CHECKPOINTS=-/CHECKPOINTS_CONVENTION.md`.

---

## 6. Правила коммитов

- Один логический документ — один коммит.
- **Код и документы не смешиваются в одном коммите.**
- Сообщения:
  - `feat(app): ...` — изменения в коде приложения.
  - `docs(phase_NN): ...` — для файлов `-=PHASES=-/Phase_NN.md`.
  - `docs(checkpoint): add YYYYMMDD_NNN ...` — для чекпоинтов.
  - `docs(tasks): ...` — для изменений в `-=tasks=-` (конвенции, планы, результаты).

---

## 7. Когда строгий цикл не обязателен

Простые документы — анализ, ретроспектива, исправление опечатки, quick-fix —
могут создаваться сразу финальными без `_plan`/`_result`/`_draft`, если в них нет
открытых вопросов для обсуждения. Решение принимает ИИ по контексту;
если сомневаешься — оформляй как `_plan` или `_draft`.

---

## 8. Отличия от исходной версии TASK_CONVENTIONS

Исходная версия предполагала фазовые директории `phase_00/`, `phase_01/` внутри
`-=tasks=-` и файлы `_draft.md`/`_final.md`. В текущем проекте исторически
сложилась датированная структура, поэтому:

- фазовые директории заменены на датированные `YYYY-MM-DD/`;
- `_draft`/`_final` оставлены как опциональный инструмент для сложных решений;
- основной цикл — `plan` → `result`;
- фазы и чекпоинты вынесены в отдельные корневые каталоги.
