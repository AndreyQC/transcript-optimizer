## 🎤 Доклад 1: «Агенты переоценённой процедурной памяти для порядка в операционном хаосе»

**Докладчик:** Михаил Шведков (фронтенд/бэкенд-разработчик, активно работает с AI-агентами, расход ~1 млрд токенов/мес. на Cursor)

### Краткое содержание

**1. Переоценённость агентов**
- Агенты хорошо справляются с творческими задачами и интерпретацией
- Плохо справляются с последовательными инструкциями: искажают, пропускают шаги, «забывают» исходную задачу
- Чем больше инструментов/MCP/скиллов у агента, тем хуже он выбирает правильный вариант и тем быстрее забивается контекстное окно

**2. Процедурная память**
- Заимствованный термин из психологии (неосознанные действия: дыхание, сон)
- Формы: **Skills**, **Workflow**, **Tools**, **Runbook**
- Скилл — наиболее явная связь с процедурной памятью (агент знает, что вызвать)

**3. Эксперимент с DeckBuddy (генератор презентаций)**
- Доклад ~15K токенов → итоговый ответ 50K токенов
- ~1/3 уходит на объявление скиллов, MCP, системные промпты
- ~40% (20K токенов) — генерация HTML-кода презентации
- Исследования подтверждают: до 60K токенов может уходить просто на описание того, «что я умею»

**4. Решение: промежуточный DSL**
- Вместо Markdown-модель генерит псевдоразметку (DSL)
- Отдельный Python-скрипт превращает DSL в HTML
- Результат: экономия ~50% токенов на генерацию, Markdown вырос с 3K до 3.2K токенов (пренебрежимо)

**5. Сравнение моделей**
- **Opus**: сожрал ~100K токенов на «подумать», запорол структуру презентации
- **Composer** и др.: плюс-минус приемлемый результат
- Цена через GPT-сото: ~$1.5 за презентацию на 10 слайдов

**6. Упомянутые исследования**
- **Hermes** — самообучающийся агент с сохранением скиллов
- Исследования 2023 г.: малые модели, которые пишут собственные скрипты, догоняют большие по эффективности
- Модели всё чаще сами генерят себе скрипты/тулы для экономии токенов

### Выводы Михаила
- Агенты полезны, но это **не универсальный исполнитель**
- Генерация человекочитаемых текстов на не-английских языках — серьёзная проблема
- Большой оверхед инструментов ухудшает выполнение даже простых задач
- DeckBuddy работает отлично «как есть», но есть что оптимизировать

### Ответы на вопросы
- **Замер токенов:** смотрел разбивку прямо в Cursor (системный промпт, MCP, скиллы, сабагентов и внутренний conversation)
- **Фреймворки для детерминированных частей:** что угодно — скрипты, **Dify**, **n8n**, **LangChain**, **LangGraph**, даже кусок Python-кода прямо в скилле; ключевое — разделить творческую и детерминированную части

## 🎤 Доклад 2: «Data Sculptor — презентация первых возможностей»

**Докладчик:** Андрей Моспанов (дата-инженер)

### Краткое содержание

**1. Задача**
- Нужно было загрузить исторические данные с 2023 г. в новые таблицы для заказчика
- ~1000 Excel-файлов с типовыми проблемами

**2. Типовые проблемы исходных файлов**
- Хаотичные имена (а дата в имени парсится в колонку → критична)
- Разное количество колонок в файлах
- Лишние колонки
- Перепутанный порядок колонок (раньше дата прихода была до даты заказа)
- Несоответствие формата дат (Excel хранит как дни с 1900 г.)
- Требовался лист «Sheet1» для корректного чтения

**3. Решение — Data Sculptor (Python + GUI)**
- Создан через **Codex** за пару дней
- Без формальной спецификации — только описание на естественном языке
- Возможности:
  - Массовое переименование файлов (с превью и масками)
  - Конвертация Excel → CSV (с настройкой кодировки и разделителя)
  - Удаление колонок (по имени)
  - Swap-колонок
  - Добавление пустой колонки с заголовком
  - Просмотр структуры всех файлов папки
  - Прогресс-бар при обработке
- GUI на русском, есть **инсталлятор (.exe)**
- Можно использовать из терминала (CLI) → можно натравить агента как утилиту

**4. Оптимизации**
- Изначально Codex читал файлы целиком → исправлено на чтение только заголовков
- Обработка батчами по 150–200 файлов

### Выводы Андрея
- Экономия времени — «куча времени» по сравнению с ручной обработкой
- Наглядность структуры всех файлов сразу → сложнее что-то пропустить
- Приложение можно переиспользовать: другие исторические загрузки точно будут

### Ответы на вопросы
- **Формулы в Excel:** в его файлах проблем не было; основная сложность — даты
- **Производительность:** не идеальна, но несопоставимо лучше ручной обработки

## ✅ Принятые решения и рекомендации

- **Не давать агенту доступа к инструментам, которые не нужны для задачи** — снижает оверхед и повышает точность выбора
- **Повторяющиеся задачи оформлять как Skill** — вместо повторного инструктирования каждый раз
- **Детерминированные части workflow выносить во внешние скрипты/сервисы** — от Dify/n8n до Python-скриптов и LangChain/LangGraph
- **Разделять творческую и детерминированную части задачи** — LLM не должна генерить детерминированный код, который может написать скрипт
- **Считать токены и смотреть их разбивку** в Cursor/IDE — для понимания реального оверхеда
- **Не отправлять внутренние/проектные данные во внешние LLM** (Cursor, Codex и пр.) — риск утечки имён, назва
