## 🎤 Доклад 1: «Агенты переоценены: процедурная память для порядка в операционном хаосе»

**Докладчик:** Михаил Шведов

**Содержание:**
- Докладчик исследует расход токенов (~1 млрд/мес.) и деградацию качества LLM-агентов на повторяющихся задачах.
- Агенты хорошо справляются с творческими и интерпретационными задачами, но плохо — с последовательными инструкциями (runbook-подобными).
- Введено понятие **процедурной памяти** агента (по аналогии с психологией — неосознанные действия: дыхание, сон).
- Формы процедурной памяти: **Skills**, **Workflows**, **Tools**, **Runbooks**.
- Эксперимент на навыке `Deckbody` (генерация презентаций): на задачу ~10–15 тыс. токенов итоговый расход составил **~50 тыс. токенов** (≈1/3 — объявления skills/MCP/системные промпты, ~40% — генерация HTML).
- Согласно исследованиям, до **60 тыс. токенов** может уходить только на описание инструментов и MCP-протокол.
- Стоимость генерации одной 10-слайдовой презентации на SOTA-модели — **~$1.5**.
- Сравнение моделей показало, что **Opus** израсходовал ~100 тыс. токенов на размышления и испортил структуру презентации; модели семейства Composer справились кратно лучше.
- **Эксперимент 1:** вынос генерации HTML в чистый скрипт без LLM — расход токенов упал до нуля, но **полная деградация качества** (потеря данных, кривая разметка).
- **Эксперимент 2 (успешный):** введён промежуточный **DSL** между Markdown и HTML; скрипт знает правила разметки, skill знает, когда какие блоки применять. Результат: расход сократился **примерно вдвое** при сопоставимом качестве (Markdown вырос с 3 000 до 3 200 токенов — пренебрежимо).
- Упомянуты подходы **Hermes** (самообучающийся агент, сохраняющий удачные skills) и исследования 2023 года, где небольшие модели с собственными скриптами догоняли SOTA по эффективности.

**Вывод:** агенты полезны, но не являются универсальными исполнителями. Критически важно разделять **творческую (семантическую)** часть (оставлять модели) и **детерминированную процедурную** часть (выносить наружу).

## 🎤 Доклад 2: «Data Sculptor — демо возможностей»

**Докладчик:** Андрей Моспанов

**Содержание:**
- Data-инженер получил от заказчика ~**1000 Excel-файлов** исторических данных для загрузки в новые таблицы.
- Проблемы: разные имена файлов (в имени закодирована дата), структура файлов со временем менялась, встречались лишние/перепутанные/отсутствующие колонки.
- Создал приложение **Data Sculptor** с помощью Codex GPT на Python за ~2 дня («наточить топор, чтобы срубить лес»).
- Возможности:
  - Массовое переименование файлов по маске (дата + латиница).
  - Просмотр структуры файлов с прогресс-баром.
  - Удаление колонок (`Delete Column`).
  - Swap колонок местами.
  - Добавление пустых колонок с заголовком.
  - Конвертация Excel → CSV с настройкой кодировки и разделителя.
- Особенности реализации:
  - GUI-оболочка (не терминал).
  - Инсталлятор (setup).
  - Переключаемый язык интерфейса (в т.ч. русский).
  - Выбор конкретного листа Excel (`Sheet1`).
  - Изначально читал весь файл — исправлено на чтение только заголовков (ускорение).
- Обработка велась батчами по 150–200 файлов. Замечание: проблемы с формулами не было, но возникали проблемы с датами в Excel-формате (число дней с 1900 г.) — функционал дописан отдельно.
- Репозиторий опубликован на GitHub (ссылка отправлена в чат).

**Вывод:** вместо того чтобы натравливать LLM на каждую пачку файлов (риск галлюцинаций, утечка данных), дешевле и надёжнее сделать узкоспециализированную GUI-утилиту. В перспективе — обернуть в CLI и подключить как Tool для агента.

📌 **Основные темы:**

- Переоценка LLM-агентов на детерминированных/последовательных задачах.
- Процедурная память агента: формы и применение (Skills, Workflows, Tools, Runbooks).
- Оверхед расхода токенов на инструменты, MCP и генерацию HTML.
- Разделение творческой и детерминированной частей задачи.
- DSL как промежуточный слой между LLM и кодогенерацией.
- Практический кейс: автоматизация обработки исторических Excel-файлов через Codex GPT.
- Безопасность данных при работе с Cursor/Codex (утечка чувствительных данных во внешние сети).

✅ **Принятые решения и рекомендации:**

- **Не давать агенту доступ к инструментам, которые не нужны для решения конкретной задачи** — это кратно снижает оверхед токенов и количество галлюцинаций.
- **Повторяющиеся задачи** (чаще «раз в никогда») оформлять как **Skill**.
- **Строго детерминированные шаги** выносить из LLM во внешний воркфлоу/скрипт; LLM оставлять только семантику.
- Использовать **промежуточный DSL** вместо прямой генерации HTML/кода в LLM — получаем и экономию токенов, и предсказуемость результата.
- Для массовой обработки структурированных файлов **создавать узкоспециализированные утилиты с GUI** (Python + библиотеки для Excel/CSV), а не «скармливать» папки в LLM напрямую.
- **Внутренние/проектные данные** не отправлять во внешние LLM (Cursor, Codex и т. п.) — риск утечки и попадания в обучающие данные.
