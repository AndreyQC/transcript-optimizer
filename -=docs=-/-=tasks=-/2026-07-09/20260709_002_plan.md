## 5. Модель данных — TypeScript-типы схем

Выведены из **фактических** файлов `sample/` (свёрено, не выдумано).

```ts
// settings.yaml
interface SettingsFile { settings: Settings }
interface Settings {
  aggressive_clean: boolean;
  similarity_threshold: number;   // 0..1
  min_word_len: number;
  suggestions_header: boolean;
}

// filler.yaml
interface FillerFile {
  filler_words: string[];
  filler_phrases: string[];
  keep_override: string[];
}

// replacements.yaml — замены + глоссарий проекта (idea.md §10.1)
interface ReplacementsFile {
  replacements: Record<string, ReplacementRule>;     // ключ replacement_rule_NNN
  lemma_replacements: Record<string, LemmaRule>;     // ключ lemma_rule_NNN
}
interface ReplacementRule {
  to: string;
  label: string;            // ОБЯЗАТЕЛЕН: id категории из glossary.yaml
  description?: string;     // опционально: определение термина (глоссарий)
  from: string[];           // всегда список, даже из одного элемента
}
interface LemmaRule {
  to: string;
  label: string;            // ОБЯЗАТЕЛЕН и для lemma-правил
  description?: string;
  from_lemmas: string[];
}

// glossary.yaml — enum категорий для label (idea.md §10.7)
interface GlossaryFile {
  categories: Record<string, GlossaryCategory>;      // ключ = id категории (snake_case)
}
interface GlossaryCategory {
  title: string;            // человекочитаемый заголовок для UI
  description?: string;
}

// lemma_irregular.yaml
interface LemmaIrregularFile { lemma_irregular: Record<string, string> }

// detector_whitelist.yaml
interface WhitelistFile { common_words: string[] }

// Дискриминированное объединение для UI
type DictFile =
  | { kind: 'settings'; data: SettingsFile }
  | { kind: 'glossary'; data: GlossaryFile }
  | { kind: 'filler'; data: FillerFile }
  | { kind: 'replacements'; data: ReplacementsFile }
  | { kind: 'lemma_irregular'; data: LemmaIrregularFile }
  | { kind: 'whitelist'; data: WhitelistFile };

// Производное: «эффективный whitelist» = common_words ∪ {все to из replacements/lemma}.
// Вычисляется в рантайме движка, НЕ пишется в detector_whitelist.yaml (idea.md §8.3.4).
function effectiveWhitelist(whitelist: WhitelistFile, replacements: ReplacementsFile): Set<string>;
```

Контракт имён файлов (автодетект по точному имени):

| Имя | `kind` |
|---|---|
| `settings.yaml` | `settings` |
| `glossary.yaml` | `glossary` |
| `filler.yaml` | `filler` |
| `replacements.yaml` | `replacements` |
| `lemma_irregular.yaml` | `lemma_irregular` |
| `detector_whitelist.yaml` | `whitelist` |

## 6. AST-правка YAML (главная техническая задача этапа 1)

Принцип (idea.md §4.4): **не строковая правка, а CST-редактирование** через пакет `yaml`.

- Парсить документ с сохранением CST (`new Document(...)` / `parse` с `keepCstNode`).
- Вставка записи «на нужное место» по типу:
  - `replacements` → новый ключ `replacement_rule_{max+1:03d}`; обязательные `to`+`label`,
    опциональное `description`, `from` всегда список. Аналогично для `lemma_replacements` (`from_lemmas`).
  - `glossary.yaml` → новый `id` категории (snake_case) с `title`/`description`.
  - `filler_words`/`common_words` → дописать строку в конец списка.
  - `keep_override` → дописать в конец списка.
  - `lemma_irregular` → новая пара `ключ: значение`.
- Сохранять стиль существующих записей (кавычки, отступы) и комментарии.
- **Перед записью** — превью diff; **после записи** — повторный парсинг как валидация. Если
  результат невалиден → откат и сообщение.
- Undo-стек по операциям над словарями.

> На этапе 1 контекстного меню по транскрипту ещё нет (нет движка). AST-правка
> задаётся через **формы редактирования словарей** (добавить/удалить запись в выбранной секции),
> что и проверяет механику CST-правки до того, как она понадобится в ПКМ-меню.

## 8. Этап 1 — задачи с критериями готовности

**1.0 Скаффолд**
- Tauri 2 + Vite + React + TS; Monaco подключён и рендерит пустой YAML.
- Tauri-команды `read_file(path)`, `write_file(path, contents)`, `pick_dir()`, `pick_file()`.
- Zustand-store с полями `dir: string|null`, `files: DictFile[]`, `ui` (тема, активная вкладка).
- Критерий: `npm run tauri dev` стартует; вручную открыть и сохранить текстовый файл.

**1.1 Автодетект и чтение**
- `detect.ts`: по выбранной директории находит 6 файлов по контракту имён, парсит, типизирует.
- Неизвестные файлы → статус «unknown» в UI (не падаем).
- Критерий: открыть `sample/transcript_optimizer` → 6 вкладок с распознанным содержимым
  (settings, glossary, filler, replacements, lemma_irregular, whitelist).

**1.2 Редактор YAML (Monaco)**
- Вкладки по словарям; Monaco в режиме `yaml`, read/write через AST-представление + raw.
- Подсветка ошибок парсинга красными волнами (Monaco markers).
- Критерий: правка видна в реальном времени; невалидный YAML подсвечивается.

**1.3 Валидация по схеме + контроль ссылочной целостности**
- JSON-Schema на каждый `kind`; ошибки → Monaco markers с позицией.
- **`label` обязателен** в каждой записи replacements/lemma_replacements; валидируется против
  enum категорий из `glossary.yaml` (referential integrity: `label` ∈ {id категорий}).
- Семантические проверки replacements (помимо схемы):
  - уникальность `to` (два правила с одинаковым `to` — ошибка);
  - `from`/`from_lemmas` не должен встречаться в двух правилах (конфликт);
  - `to` одного правила не должен быть `from`/`from_lemmas` другого (цепочка);
  - дубликаты внутри одного `from` — предупреждение.
- Удаление категории из `glossary.yaml` блокируется, если на неё ссылается хотя бы одна запись.
- Критерий: удалить обязательное поле / ввести несуществующий `label` → маркер на позиции;
  легенда категорий ошибок; конфликт `from` подсвечивается.

**1.4 AST-правка через формы** *(проверяет механику до ПКМ-меню)*
- Формы добавления/удаления записи по секции (выбор секции → ввод значения → предпросмотр diff → сохранить).
- Соблюдение правил из §6 (нумерация `replacement_rule_NNN`, `from` всегда список, стиль/комментарии сохраняются).
- Для replacements: выбор `label` из dropdown категорий (из `glossary.yaml`), опциональное `description`.
- Критерий: добавить `replacement_rule_006` и слово в `filler_words` через форму → файл валиден, комментарии и стиль старых записей на месте; повторное чтение возвращает то же.

**1.5 Сохранение и отмена**
- Сохранение на диск через `write_file`; авто-валидация после записи; Undo для операций форм.
- Грязный-флаг (`*` в заголовке вкладки при несохранённых изменениях); подтверждение перед закрытием.
- Критерий: правка → сохранить → reopen = исходное состояние новых данных; отменить добавление записи возвращает файл.

**1.6 Экспорт Markdown-глоссария**
- Генерация `GLOSSARY.md` из `replacements.yaml` + `glossary.yaml`: термины, сгруппированные по
  категории (`label` → `title`), с `description` и списком вариантов написания (`from`/`from_lemmas`).
- Одно нажатие в UI; файл пишется в корень открытой директории (или по выбору).
- Критерий: на `sample/` экспорт даёт `GLOSSARY.md` с разделами по категориям; открыть в любом
  Markdown-просмотрщике — читаемо.

## 9. Риски / вопросы для этапа 1

1. **CST-правка пакетом `yaml`** — самая рисковая часть. Решение: сначала сделать отдельный
   спайк — прогнать на `replacements.yaml` сценарий «добавить правило» и убедиться, что
   комментарии/стиль/порядок сохраняются. Если пакет не справится — fallback к ручному
   текстовому редактированию с строгой валидацией (менее надёжно, но работает).
2. **Кодировки/переносы** — файлы в UTF-8, переносы CRLF/LF сохранить как в исходнике.
3. **Размер `detector_whitelist.yaml`** (~580 слов) — Monaco справится; валидация только по схеме
   списка строк, без проверки дублей (дубли — предупреждение, не ошибка).
