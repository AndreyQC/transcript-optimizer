# Phases — conventions


Назначение `-=PHASES=-`: дать **высокоуровневый свод по завершённой фазе** —
что входило в фазу, что сделано, какие ключевые решения приняты, что осталось
за рамками. Файл фазы читают, когда нужно понять итог этапа целиком, не
погружаясь в детали планов и результатов из `-=tasks=-`.

---

## 1. Когда пишется Phase-файл

- **Всегда — по завершении фазы.** Фаза считается завершённой, когда её задачи
  реализованы, проверены и залиты в `main`.
- **Один файл на фазу.** Не дробите фазу на несколько `Phase_NN_*.md`; если
  фаза разбита на подэтапы — отражайте их разделами внутри одного файла.
- **Не пишется для незавершённой работы.** Текущее состояние незакрытой фазы
  живёт в чекпоинте (`-=CHECKPOINTS=-`), а план/результат — в `-=tasks=-`.

---

## 2. Именование файлов

```
Phase_NN.md
```

- `NN` — двузначный порядковый номер фазы: `Phase_01`, `Phase_02`, …, `Phase_08`.
- Номер **сквозной по проекту**, не сбрасывается по годам/датам.
- Никаких суффиксов `_v2`, `_final`, `_draft` — фаза либо завершена (один файл),
  либо нет (файла ещё нет). Правки в опубликованный файл вносятся через
  новый коммит (`docs(phase_NN): ...`), а не через дублирование.

---

## 3. Структура (разделы)

Файл фазы — самодостаточный документ, читаемый без контекста чата.
Рекомендуемые разделы (пропускайте только если раздел действительно пуст):

1. **Заголовок H1** — `Phase N: <краткое название фазы>`.
2. **Метаблок** вверху: `Дата`, `Статус` (завершён), ссылка на `План` и (если
   есть) на `Результат` из `-=tasks=-`.
3. **Цель фазы** — 2–5 пунктов: что должно было стать результатом фазы.
4. **Что сделано** — таблицы по категориям (`Backend`, `Frontend`, `Sample-данные`,
   …), колонки «Файл → Что изменилось». Конкретные пути относительно корня репо.
5. **Ключевые архитектурные решения** — bullet-список неочевидных выборов,
   которые стоит знать следующей сессии (почему так, а не иначе; чем закрыта
   дыра из `LESSONS_LEARNED.md`; какие ограничения приняты осознанно).
6. **Проверки** — точные команды и их результат (`OK`, `N passed`),
   включая ручные/временные проверки.
7. **Известные ограничения / NOT done** — явный список того, что не сделано или
   отложено. Честность здесь экономит время следующей сессии.
8. **Где читать дальше** — упорядоченный список ссылок (план, результат,
   предыдущая фаза, релевантные разделы `LESSONS_LEARNED.md`).

Разделы 4 (таблицы «что сделано») и 7 (NOT done) — самые ценные; держите их
аккуратными и конкретными.

---

## 4. Правила написания

- **Высокоуровневый свод, не дубликат.** Phase-файл обобщает, а не пересказывает
  `-=tasks=-` дословно. Детали реализации — в plan/result; здесь — итог и связи.
- **Конкретные пути** — всегда относительно корня репо
  (`app\src\engine\rules.ts`, не «движок замен»).
- **Хэши коммитов / ссылки** — для проверяемости (`git show <hash>`).
- **Без нового дизайна.** Фаза описывает *что сделано*, а не *что следовало бы*.
  Открытые вопросы и проработки — в `-=tasks=-/..._draft.md`.
- **Кросс-ссылки вместо дублирования.** Если решение разобрано в
  `LESSONS_LEARNED.md` §6 — сошлитесь, не пересказывайте.
- **Простой markdown, ASCII-таблицы, без эмодзи.** Любой вьюер, чистый diff.
- **Русский текст, английские технические идентификаторы** — не смешивать
  в одном предложении.

---

## 5. Объём

Ориентир — **80–130 строк**. Если выходит длиннее — таблицы §4, скорее всего,
дублируют содержимое `-=tasks=-`: замените пересказ ссылками. Если короче —
вероятно, неполон раздел §4 или §7.

---

## 6. Коммит

Один коммит на файл фазы (код и документы не смешивать — см.
`TASK_CONVENTIONS.md` §6):

```
docs(phase_08): фразовые замены + числа не OOV + категории глоссария
```

---

## 7. Связь с `-=tasks=-` и `-=CHECKPOINTS=-`

| | `-=tasks=-` | `-=PHASES=-` | `-=CHECKPOINTS=-` |
|---|---|---|---|
| Что хранит | Детальная история (план, результат, draft) | Итоговый свод по фазе | Снапшот «где мы сейчас» |
| Гранулярность | Одна задача — пара документов | Одна фаза — один файл | Один момент — один файл |
| Когда читают | Проектируя следующий шаг | Понимая итог завершённого этапа | Первым делом, каждой новой сессией |
| Дублирование | Источник деталей | Ссылается на tasks | Ссылается на tasks и phases |

Порядок чтения для новой сессии:
**последний чекпоинт** → **фаза, на которой стоит работа** → при необходимости
**конкретные task-файлы** этой фазы.
