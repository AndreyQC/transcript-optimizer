# Lessons Learned — ошибки запуска desktop-приложения (Tauri 2 + React + Zustand + Vite)

Документ фиксирует реальные проблемы, возникшие при первом запуске приложения, и
общие правила, как их избегать в других проектах на этом стеке. Ошибки упорядочены
по времени проявления — от установки до рантайма.

---

## 1. pnpm: невалидный `pnpm-workspace.yaml` роняет `pnpm install`

### Что произошло
`pnpm create tauri-app` оставил шаблонный ключ в `pnpm-workspace.yaml`:
```yaml
allowBuilds:
  esbuild: set this to true or false   # плейсхолдер, НЕ валидное значение
```
Позже рядом был добавлен «правильный» ключ `onlyBuiltDependencies`, но шаблонный
не удалили. pnpm 11.10 в такой конфигурации блокирует ВСЕ build-скрипты и падает с
`ERR_PNPM_IGNORED_BUILDS`, что ломало pre-flight `pnpm install` внутри `tauri dev`.

### Почему это опасно
- `tauri dev` запускает `pnpm install` перед стартом — ошибка на этом шаге
  выглядит как «Tauri не работает», хотя корень в пакетном менеджере.
- Шаблонные плейсхолдеры легко не заметить: `set this to true or false` читается
  как комментарий, но парсится как строка-значение.

### Как избегать
- **После любого `create-*` скаффолда ревьюить сгенерированные конфиги целиком**,
  особенно файлы с плейсхолдерами (`pnpm-workspace.yaml`, `.env.example`). Не
  оставлять «шумные» ключи рядом с рабочими.
- В pnpm 11.x список пакетов, которым разрешён postinstall, задаётся через:
  ```yaml
  # pnpm-workspace.yaml (формат pnpm 11.10)
  allowBuilds:
    esbuild: true
  ```
  Или эквивалент `onlyBuiltDependencies` — но **не оба ключа одновременно** и
  **не с шаблонным значением**.
- Проверять, что build-скрипт реально отработал: после `pnpm install` платформенный
  бинарь должен присутствовать (`ls node_modules/.pnpm/@esbuild/win32-x64*/...`).
  Vite, падающий с `RC=127` (command not found), — признак, что esbuild-postinstall
  не выполнился.
- Тест запуска: `pnpm install` должен выходить с **RC=0** без `[ERR_PNPM_*]`.

## 2. Tauri + Vite: пустое окно из-за IPv4/IPv6 (Windows)

### Что произошло
Шаблонный `vite.config.ts` использует `host: host || false`. На Windows WebView2
иногда резолвит `localhost` в IPv6 (`::1`), а Vite с `host: false` слушает на
другом интерфейсе — соединение молча падает. Окно остаётся белым, DevTools-консоль
пустая, ошибок нет.

### Почему это опасно
- **Полное отсутствие диагностики**: белое окно + пустая консоль выглядят как
  «вообще ничего не работает», хотя проблема в сетевом биндинге.
- Симптом неотличим от рантайм-JS-ошибки без отдельной проверки.

### Как избегать
- Биндить Vite на явный адрес, совпадающий с `devUrl`:
  ```ts
  // vite.config.ts
  server: {
    port: 1420,
    strictPort: true,
    host: host || "127.0.0.1",   // явно IPv4, не false
    // ...
  }
  ```
- **Диагностический приём для белого Tauri-окна**: открыть `http://127.0.0.1:1420`
  в системном Edge (тот же движок, что WebView2).
  - Если UI есть в Edge, но нет в Tauri → проблема WebView2/окружения.
  - Если пусто и в Edge → проблема во фронтенде (Vite/React/модули).
  - Если страница не грузится в Edge → проблема в dev-сервере (порт/биндинг).
- Заголовки Network-запросов помогают отделить слои: `origin`/`referer` показывают,
  грузится ли страница вообще; `user-agent` подсказывает, в каком браузере/WebView.

## 3. Zustand + React 18+: бесконечный ре-рендер от «некэшированного» селектора

> **Главная ошибка этой сессии.** Белое окно в итоге объяснилось именно ей.

### Что произошло
В store был селектор `selectGlossaryCategories(state)`, возвращающий **новый `Set`**
при каждом вызове:
```ts
// ❌ ОПАСНО: новый объект каждый раз
export function selectGlossaryCategories(state): Set<string> {
  const cats = new Set<string>();
  // ... заполнение ...
  return cats;
}
// в компоненте:
const cats = useDictionaries(selectGlossaryCategories);
```
Под капотом zustand использует `useSyncExternalStore`, который сравнивает снимки
через `Object.is`. `Set` всегда «новый» по ссылке → React решает, что состояние
изменилось → ре-рендер → селектор снова → новый `Set` → снова «изменилось» →
**бесконечный цикл**. React падает с:
```
The result of getSnapshot should be cached to avoid an infinite loop
Maximum update depth exceeded
```
Выброшенная ошибка убивала первый рендер → окно оставалось белым.

### Почему это опасно
- Ошибка несовместима с «явными» симптомами: белый экран + (в WebView2) пустая
  консоль. Создаёт ложное впечатление проблемы с окружением.
- Проявляется только при подписке через `useStore(selector)` — в unit-тестах,
  где селектор зовётся вручную, всё работает (Set валиден, валидация проходит).
  Smoke-тесты давали зелёный свет, а в React-рендере всё падало.
- В React 18 `useSyncExternalStore` стал строже к кэшированию снимков, чем в 17 —
  код, работавший раньше, может suddenly сломаться.

### Правило
**Zustand-селектор никогда не должен возвращать новый объект/массив/Set/Map при
каждом вызове.** `useSyncExternalStore` сравнивает результат по ссылке (`Object.is`);
новая ссылка = «состояние изменилось» = ре-рендер.

| Тип возврата | Безопасно? |
|---|---|
| примитив (`string`, `number`, `boolean`) | ✅ сравнивается по значению |
| стабильная ссылка (поле store, элемент массива через `find`) | ✅ ссылка та же между рендерами |
| `null` / `undefined` | ✅ |
| `new Object()` / `{ ...spread }` | ❌ новая ссылка каждый раз |
| `new Array()` / `.filter()` / `.map()` / `.slice()` | ❌ |
| `new Set()` / `new Map()` | ❌ |
| отсортированный/изменённый массив | ❌ |

### Как избегать (3 паттерна)

**Паттерн A — вычислять в `useMemo` от стабильной подписки (использован здесь):**
```ts
// подписываемся на стабильную ссылку (массив entries из store)
const entries = useDictionaries((s) => s.entries);
// производную коллекцию мемоизируем
const cats = useMemo(() => new Set(extractCategories(entries)), [entries]);
```

**Паттерн B — `useShallow` для объектов/массивов, если нужна подписка сразу на
несколько полей:**
```ts
import { useShallow } from "zustand/react/shallow";
const { foo, bar } = useStore(useShallow((s) => ({ foo: s.foo, bar: s.bar })));
```

**Паттерн C — выносить производные селекторы в отдельный хук с собственным
мемо-кэшем** (для сложных случаев).

### Признаки, что вы наступили на эту ошибку
- `Maximum update depth exceeded` / `getSnapshot should be cached`.
- Белый экран, особенно в production-сборке (в dev React иногда ловит и логирует).
- UI работает в unit-тестах (где нет подписки), но падает в React-рендере.
- В DevTools — avalanche ре-рендеров одного компонента.

### Аудит других селекторов
После фикса стоит проверить ВСЕ `useStore(selector)` в проекте. Безопасны
селекторы, возвращающие:
- примитивы (`boolean`, `number`, `string`);
- `find()`/прямое обращение к существующему элементу (та же ссылка);
- поля-ссылки из store (`s.entries`, `s.data`), если они не пересоздаются в каждом
  `set` без необходимости.

Опасны — любые `.map()`/`.filter()`/`new Set()`/спреды `{...}`/`[...]` внутри
селектора.

## 4. Tauri: `onCloseRequested` с async-диалогом может заблокировать закрытие

### Что произошло
Обработчик закрытия окна вызывал `ask()` (Tauri dialog), и если диалог выбрасывал,
`preventDefault()` оставался «висящим» — окно не закрывалось, кнопка закрытия не
реагировала.

### Как избегать
- **Окно обязано закрываться всегда**, кроме явного подтверждения. Асинхронные
  операции в close-handler — в try/catch, при ошибке — НЕ блокировать закрытие:
  ```ts
  win.onCloseRequested(async (event) => {
    try {
      if (!hasUnsavedChanges()) return;
      const ok = await ask("Закрыть без сохранения?");
      if (!ok) event.preventDefault();
    } catch (e) {
      console.error("close handler failed, allowing close:", e);
      // НЕ вызываем preventDefault — даём окну закрыться
    }
  });
  ```
- Внешний try/catch — для случая «не Tauri-окружение» (`getCurrentWindow()`
  выбрасывает в обычном браузере).

## 5. Общий методологический урок: разделяй слои при диагностике белого окна

«Пустое окно» в desktop-приложении может означать поломку на ЛЮБОМ из уровней, и
без разделения слоёв легко чинить не то. Порядок проверки:

1. **Пакетный менеджер** — `pnpm install` выходит RC=0? Нет `ERR_PNPM_*`?
2. **Build-скрипты** — platform-бинари на месте (`@esbuild/*`)? `vite build`/`tsc` чистые?
3. **Dev-сервер** — Vite стартует на ожидаемом порту? Совпадает ли адрес с `devUrl`?
4. **Сеть до сервера** — страница грузится в системном Edge по тому же URL?
5. **Фронтенд-рантайм** — есть ли ошибки в DevTools Console (именно в Edge, не
   только в WebView2 — там виднее)?
6. **Tauri-окружение** — плагины зарегистрированы, capabilities выданы?

Каждый шаг сужает гипотезу. Главное — **не пытаться чинить всё сразу**, а
изолировать слой. В этой сессии шумные запросы антивируса в Network-вкладке
сбивали с толку, пока проверка `origin`/`referer` не подтвердила, что страница
грузится (слой 4 ОК) — значит проблема во фронтенд-рантайме (слой 5).

## 6. Многословные правила и OOV-фильтрация

**Контекст.** Движок исключает OOV-токены через три источника:

1. `effectiveWhitelist` (`common_words` ∪ `replacements[*].to` ∪ `lemma[*].to`).
2. `replaceIdx` — нормализованный `from` одиночных замен.
3. `fillerWords` — одиночные `filler_words` (без `keep_override`).

**Дыра:** многословные фразы из `filler_phrases` и `replacements[*].from` с
≥ 2 слов **не** покрываются ни (2), ни (3). Причина — движок удаляет их как
подстроки (`filler_phrases`) или пытается токенизировать фразу целиком и промахивается
(`replacements` — `tokenize` режет по словам). В результате токены внутри таких
фраз проходят на шаг §4 (OOV-проверка по `origTokens`), не матчатся ни через
`replaceIdx.has(key)`, ни через `fillerWords.has(key)` — и попадают в
OOV-статистику.

**Решение.** Скрывать токен из грида OOV, если **ВСЕ** его вхождения в исходном
транскрипте лежат внутри какой-либо многословной фразы из `filler_phrases` или
`replacements[*].from` (≥ 2 слов после нормализации). Критерий утверждён
пользователем 2026-07-15: «если токен OOV встречается только в многословной
фразе из filler или replacement — не выводить в статистике».

**Где фильтровать — в `buildOovRows`, НЕ в движке.** Движок остаётся источником
истины для decorations; фразовая фильтрация — это статический анализ текста +
правил, она ложится на агрегатор грида. Так:
- decorations продолжают быть честными (показывают токен как есть);
- логика движка не размывается «смежной» заботой о UI;
- unit-style тесты `applyRules` не меняются.

**Алгоритм:**
1. Собрать `Map<lineNo, [{start,end}, …]>` — все вхождения многословных фраз
   в каждой реплике (по `utt.text`, без префикса `[HH:MM:SS]`).
2. Собрать `Map<norm, [{lineNo, start, end}, …]>` — все вхождения каждого токена
   (тот же `tokenize`, что и движок — нормализация и границы слов совпадают).
3. Для каждого `key ∈ OovRow.map.keys()`: если **каждое** его вхождение
   полностью ⊂ какому-то `phraseSpan` на его `lineNo` → удалить из map.

**Нюансы, на которые стоит обратить внимание:**
- Использовать тот же `tokenize`/`norm`, что и движок, и ту же `wordBoundaryRe`
  с Unicode property escapes — иначе границы слов на кириллице разойдутся.
- Работать по `utt.text` (после парсера), не по `transcript.raw` — избегаем
  хрупкости с CRLF и префиксом `[HH:MM:SS]`.
- Перекрытие фраз («в общем» и «в общем-то») не проблема: `isInsideAny` через
  `some` ответит «да, токен внутри» для любой подходящей фразы; ложного
  сокрытия не будет.
- Пустой `ctx` или отсутствие `transcript` → поведение как раньше (ничего
  не скрываем). Это сохраняет обратную совместимость unit-вызовов.

**Урок:** если движок и UI-агрегатор используют разные правила отбора — пробелы
в движке (многословные фразы) проявляются в UI как «фантомные» OOV. Чинить
такое надо в агрегаторе, а не патчить движок костылями — иначе decorations
перестанут отражать реальность правил.

## Шпаргалка: быстрые проверки при «белом окне Tauri»

| Симптом | Скорее всего | Проверка |
|---|---|---|
| `pnpm tauri dev` падает до окна | pnpm/build-скрипты | `pnpm install` RC=0? |
| Окно открывается, но белое, консоль пуста | IPv6/IPv4 или dev-сервер | открыть `127.0.0.1:1420` в Edge |
| Белое окно, в Edge-DevTools `Maximum update depth` | некэшированный селектор | аудит `useStore(selector)` |
| Окно белое, в консоли Tauri-API TypeError | Tauri-импорт вне try/catch | обернуть в try/catch или `isTauri`-guard |
| Кнопка закрытия не работает | `onCloseRequested` блокирует | try/catch, не звать `preventDefault` при ошибке |
