**Дата:** 2026-07-09
**Статус:** Идея / анализ
**Автор:** andrey potapov (концепция), предложения — ZCode

---

## 1. Понимание задачи

Desktop-приложение для интерактивной оптимизации транскриптов (расшифровок речи) и сопровождения
YAML-словарей. Словари используются в дальнейшем пайплайне автоматической подготовки summary.
Приложение совмещает три роли:

1. **Рабочее место корректора** — оригинальный транскрипт слева, очищенный справа, статистика замен.
2. **Редактор словарей** — наполнение/валидация YAML-файлов справочников; пополнение «в один клик» из
   контекстного меню по выделенному слову.
3. **Пусковая установка для LLM-summary** — открытие prompt-файла, запуск OpenAI-совместимого LLM,
   просмотр результата в Markdown (включая таблицы и Mermaid).

## 2. Анализ существующих YAML-справочников (`sample/transcript_optimizer`)

Изучены 5 файлов. Ниже — их роль и замечания, влияющие на архитектуру приложения.

### 2.1. `settings.yaml`
Глобальные настройки пайплайна. Текущие поля:
- `aggressive_clean: false` — агрессивный режим очистки.
- `similarity_threshold: 0.65` — порог нечёткого сравнения (вероятно для подбора замен/детекции).
- `min_word_len: 3` — минимальная длина слова (для детекции «мусора»).
- `suggestions_header: true` — включить шапку блока подсказок в выводе.

**Замечание:** стоит явно задокументировать семантику каждого поля (что именно сравнивается на
`similarity_threshold`), т.к. от этого зависит поведение детектора «нераспознанных» слов в UI.

### 2.2. `filler.yaml` — слова/фразы-заполнители и междометия
Секции: `filler_words` (≈50 слов: «ага», «блин», «короче», «ща», «ээ» …), `filler_phrases` (≈30 фраз:
«как бы то ни было», «по большому счёту» …), `keep_override: []`.
Это тот самый справочник «удаление междометий и повторов».

**Замечания:**
- Слова записаны в разных регистрах и вариантах (`вроде` / `вродебы`, `как` / `какбы`, `ща` / `щас`).
  В UI нужно показать «источник попадания» (по какому правилу удалено), иначе пользователь не поймёт,
  почему исчезло слово.
- `keep_override` — важный «исключающий» список (слова, которые НЕ удалять, даже если совпали).
  Контекстное меню «отправить в словарь» должно уметь класть слово именно сюда (анти-действие).

### 2.3. `replacements.yaml` — правила замены
Две подсекции:
- `replacements` — именованные правила (`replacement_rule_NNN`) с `to`, `from[]`, необязательным
  `label`. Пример: «мини Макс / минимакс / mini max → Minimax».
- `lemma_replacements` — замены по леммам (`from_lemmas[]` → `to`), например «слак → Slack».

**Замечания:**
- Ключи `replacement_rule_NNN` — нумерованные. При «добавлении в один клик» нужно гарантировать
  уникальность и сохранять порядок/нумерацию (см. п. 4.4).
- `label` — опциональный, но ценный для группировки в UI статистики («LLM vendor», «JIRA» и т.п.).
  Стоит сделать его обязательным или автогенерируемым.
- Логически `lemma_replacements` отделены от `replacements` — это разные стадии пайплайна.
  В контекстном меню надо различать «добавить замену» vs «добавить лемматизационную замену».

### 2.4. `lemma_irregular.yaml`
Карта «слово во Flexии → лемма» (люди→человек, шли→идти, детей→ребёнок …). Используется
лемматизатором для приведения словоформ к нормальной форме перед сопоставлением со словарями.

**Замечание:** по структуре это простой `dict`. Пополнять вручную через UI можно, но проще всего
именно тут — точечные правки. Хороший кандидат на «быстрое добавление».

### 2.5. `detector_whitelist.yaml`
`common_words` — большой (~580) белый список частых слов. Слова, НЕ попавшие сюда и не прошедшие
другие проверки, попадают в детектор «нераспознанных» и подсвечиваются в оригинале.

**Замечания:**
- Список неотсортирован логически (видна ручная правка: `codereview` в начале, потом «а», потом
  «автоматизации»). Для UI нужна стабильная сортировка (по алфавиту) только для отображения, без
  изменения физического порядка в файле, либо наоборот — поддерживать сортированный файл.
- Именно сюда чаще всего будет отправляться «распознанное, но новое» слово через ПКМ.

### 2.6. Вывод по словарям
Все 5 файлов имеют **разную топологию** (список строк / список объектов / dict / секции в одном
файле). Это ключевой фактор архитектуры: приложение должно знать **схему каждого справочника**, чтобы
корректно вставлять записи «на нужное место» и сохранять валидный YAML.

## 3. Предлагаемый технический стек

Приоритет — open-source компоненты, богатая подсветка/редактирование, поддержка Markdown+Mermaid.

### 3.1. Платформа (shell)
| Вариант | Плюсы | Минусы | Оценка |
|---|---|---|---|
| **Tauri** (Rust + Web) | Маленький бинарь (~10 МБ), низкое потребление RAM, безопасность, быстрая загрузка локальных файлов через FS-API | Rust для бизнес-логики (если не выносить её в JS); моложе Electron | ★ Рекомендую |
| Electron | Огромная экосистема, много готовых решений | ~150 МБ RAM+, тяжёлый дистрибутив | Приемлемо |
| Wails (Go + Web) | Go вместо Rust, простая интеграция с CLI/пайплайном | Меньше сообщество | Если логику лучше писать на Go |

**Рекомендация:** **Tauri** — он лучше всего подходит под «оболочка над файлами + веб-редакторы»,
а основная бизнес-логика очистки, скорее всего, уже существует как отдельный пайплайн (Python?).
Приложение выступает UI-надстройкой, дёргающей либо локальный CLI/библиотеку, либо встроенную
реализацию правил.

### 3.2. Редакторы (open-source)
- **YAML и текст транскрипта:** **Monaco Editor** (от VS Code). Отличная подсветка YAML/Markdown,
  decorations API — идеально для подсветки «нераспознанных» слов (цветные подчёркивания/фоны),
  диапазоны для diff-подсветки, контекстное меню по правому клику. Либо **CodeMirror 6** (легче,
  модульнее, лучше для встраивания нескольких панелей).
- Для **бок-о-бок сравнения** (оригинал/очищенный) — Monaco `diffEditor` даёт готовый
  визуальный diff; либо две панели со связанной прокруткой.

### 3.3. Просмотр Markdown (таблицы + Mermaid)
- **react-markdown** + `remark-gfm` (таблицы, task-list, strikethrough) + `rehype-raw` (inline HTML).
- **mermaid** — рендер диаграмм из code-fence ` ```mermaid `.
- Альтернатива: **markdown-it** + плагины (`markdown-it-mermaid`).
- Бонус: режим «исходник ↔ превью» (split) для prompt-файла summary.

### 3.4. LLM-интеграция
- HTTP-клиент к **OpenAI-совместимому** Chat Completions API (`/v1/chat/completions`). Это позволит
  подключать не только OpenAI, но и совместимые эндпоинты (Azure OpenAI, локальные/прокси).
- Настройки (`base_url`, `api_key`, `model`, `temperature`, `max_tokens`, `system_prompt`) —
  хранить в отдельном конфиге (НЕ в `settings.yaml`, чтобы не утекал ключ в репозиторий).
- Потоковый вывод (SSE) — summary приходит по частям, сразу рендерится в Markdown-просмотр.

## 4. Архитектура и функциональные предложения

### 4.1. Модель директории-проекта
Пользователь открывает **директорию словарей** (как сейчас `sample/transcript_optimizer`).
Приложение определяет схему по **имени файла + по структуре**. Надо зафиксировать контракт имён:

| Файл | Роль | Тип содержимого |
|---|---|---|
| `settings.yaml` | настройки | плоский конфиг |
| `filler.yaml` | удаление filler/междометий | слова + фразы + override |
| `replacements.yaml` | замены | именованные правила + lemma-правила |
| `lemma_irregular.yaml` | карта Flexия→лемма | dict |
| `detector_whitelist.yaml` | белый список детектора | список слов |

**Предложение:** сделать эти имена конфигурируемыми (defaults + override), чтобы поддержать
несколько пайплайнов с разными именами. В UI — панель «Словари» со списком обнаруженных файлов и
статусом «распознан / неизвестная схема».

### 4.2. Трёхпанельный (или четырёхпанельный) layout
```
┌─────────────────────────────────────────────────────────┐
│ Toolbar: [Открыть папку словарей] [Открыть транскрипт]   │
│          [Открыть prompt] [Запустить summary ▶]          │
├──────────────────────┬──────────────────────────────────┤
│  Оригинал            │  Очищенный                        │
│  (Monaco, readonly   │  (Monaco,                         │
│   или редактируемый) │   diff vs оригинал)               │
│  ⚠ подсветка         │                                   │
│    нераспознанных    │                                   │
├──────────────────────┴──────────────────────────────────┤
│  Статистика замен (таблица: слово → замена, источник,    │
│   правило, сколько раз)            [фильтр/экспорт]      │
├─────────────────────────────────────────────────────────┤
│  Вкладки: [Словари ▼] filler | replacements | whitelist  │
│           | lemma  —  встроенный Monaco YAML-редактор     │
└─────────────────────────────────────────────────────────┘
```
Окно Markdown-summary открывается отдельно/вкладкой с split (исходник prompt | результат).

### 4.3. Детектор «нераспознанных» слов и подсветка
- После применения правил формируется множество «проблемных» токенов: не из whitelist, не совпало
  с заменами, ниже порога `similarity_threshold`, короче `min_word_len`, неEnglish-мусор и т.п.
- В **оригинале** они подсвечиваются ( Monaco `IModelDecoration`, цвет по категории:
  «мусор», «возможная опечатка», «неизвестное имя собственное»).
- При клике/выделении — в **правой панели** та же позиция подсвечивается (как в оригинале, так и
  в очищенной версии, если слово пережило очистку).
- **Контекстное меню (ПКМ)** по выделению предлагает целевые действия с автоматическим выбором
  файла-словаря:
  - «Добавить в whitelist» → `detector_whitelist.yaml` (если слово корректное).
  - «Удалить всегда (filler)» → `filler.yaml → filler_words`.
  - «Заменить на…» → `replacements.yaml` (с вводом замены и, опционально, label).
  - «Сохранить лемму» → `lemma_irregular.yaml`.
  - «Не трогать (keep)» → `filler.yaml → keep_override`.

### 4.4. Безопасное добавление «на нужное место» в YAML
Главная техническая сложность — вставить запись, **сохранив структуру и валидность**. Подход:
- Не «строковая» правка, а **AST-редактирование**: парсить YAML в объект с сохранением позиции
  (`yaml` npm-пакет с `keepCstNode` / CST, или Rust `yaml-rust2`/`yaml-edit`).
- Вставлять новый элемент:
  - в `replacements` — новый ключ `replacement_rule_{max+1:03d}` с шаблоном `from: [<слово>]`,
    `to: <...>`, сохраняя стиль (двойные кавычки/без, отступы) существующих записей;
  - в `filler_words` / `common_words` — добавить строку в конец списка (или с учётом `keep_override`);
  - в `lemma_irregular` — добавить пару `ключ: значение`.
- **Предпросмотр изменения** перед записью (mini-diff), затем сохранение на диск + авто-валидация
  (повторный парсинг). Если результат невалиден — откат и сообщение.
- Источник правок логируется (для Undo-стека по операциям над словарями).

### 4.5. Статистика
Таблица: `исходное слово | чем заменено | тип (replace/lemma/filler-удаление) | правило | count`.
- Сортировка по частоте; фильтр по типу/файлу-источнику; экспорт в CSV/Markdown.
- Сводка сверху: всего слов, заменено N, удалено M, подозрительных K.

### 4.6. Запуск summary
- Пользователь открывает **prompt-файл** (Markdown/текст) — отображается в split-редакторе.
- Кнопка «Запустить» отправляет: системный промпт + очищенный транскрипт + шаблон prompt → LLM.
- Результат рендерится в Markdown-просмотр (GFM + Mermaid). Кнопка «Сохранить .md».
- История запусков (prompt-хэш, модель, токены, дата) — для воспроизведения.

## 5. Дополнительные предложения (улучшения сверх ТЗ)

1. **Превью пайплайна в реальном времени.** При правке любого словаря — авто-пересчёт очищенного
   транскрипта (debounce ~300 мс), чтобы пользователь сразу видел эффект.
2. **Версионирование словарей.** Лёгкая интеграция с git прямо из приложения (commit изменений
   словарей одной кнопкой), т.к. YAML-файлы — это и есть артефакт ценности.
3. **Схема/валидация словарей.** JSON-Schema на каждый файл; подсветка ошибок прямо в Monaco
   (красные волны под невалидными ключами). Защита от поломки пайплайна «кривой» правкой.
4. **Тестовые транскрипты.** Набор мини-примеров (fixtures) с эталонным результатом — regression-тесты
   при изменении словарей: «сломалась ли замена X после правки».
5. **Горячие клавиши** для частых действий: `F2` — переименовать везде, `Ctrl+Enter` — запустить
   summary, `Ctrl+Shift+W` — добавить в whitelist.
6. **Мультиязычность интерфейса** (RU/EN) с самого начала — словари русскоязычные, но UI может
   понадобиться англоязычному контрибьютору.
7. **Тёмная тема** (Monaco и Markdown-просмотр её поддерживают).
8. **Офлайн-работа.** Всё, кроме вызова LLM, должно работать без сети.

## 6. Риски и открытые вопросы

1. **Где живёт логика очистки?** Сейчас YAML-файлы — это данные для некоего пайплайна. Нужно
   понимать: приложение (а) только редактирует словари и показывает стат. по результатам внешнего
   шага, либо (б) само реализует правила очистки? От этого радикально зависит объём работ.
   → **Рекомендация:** реализовать движок правил в самом приложении (JS/Rust), чтобы давал
   интерактивное превью; при этом поддержать режим «импорт готового отчёта» от внешнего пайплайна.
2. **Формат транскрипта.** ~~Открыт вопрос закрыт~~ — см. новый раздел 8 «Анализ реального
   транскрипта». Формат — плоский текст (`.txt`) с хидерами спикеров и тайм-штампами `[HH:MM:SS]`.
3. **Лемматизатор.** Для `lemma_replacements` и `lemma_irregular` нужен стабильный лемматизатор
   (pymorphy3/спец). Встраивать его в приложение или доверять предвычисленным леммам из отчёта?
4. **Безопасное хранение API-ключа.** OS keychain (Windows Credential Manager) — обязательно,
   не plaintext и не в репозитории.

## 7. Рекомендуемый первый шаг (MVP)

Чтобы быстро получить ценность, предлагаю **MVP в 4 этапа**:

1. **Каркас + открытие словарей.** Tauri + Monaco; чтение/валидация/редактирование 5 YAML-файлов
   с подсветкой и JSON-Schema-валидацией.
2. **Очистка транскрипта (движок правил).** Минимальный движок (filler + replacements + whitelist),
   двухпанельный просмотр с подсветкой подозрительных слов и таблицей статистики.
3. **Контекстное меню → добавление в словарь** с AST-правкой YAML и превью diff.
4. **LLM-summary.** Prompt-редактор + вызов OpenAI-совместимого API + Markdown(Mermaid)-просмотр.

Каждый этап самостоятелен и приносит пользу: даже (1) уже удобнее ручной правки YAML в редакторе.

## 8. Анализ реального транскрипта (образец)

Изучен образец `sample/transcript_optimizer/2026-06-30 14_03 (MSK) Drill_AI_Skill.2026 [0FSklp].txt`
(731 строка, ~45 000 символов, ~49 мин записи). Раздел закрывает прежний открытый вопрос 6.2 и
конкретизирует требования к парсеру и подсветке.

### 8.1. Формат файла
Плоский текст (`.txt`), **не** JSON/SRT. Структура строки:
- Заголовок-шапка (2 строки): дата, время, дисклеймер про неточности.
- **Блок спикера:** `Имя Фамилия (голос N):` — имя может быть с уточнением голоса
  (`Andrey Potapov (голос 1)`, `Andrey Potapov (голос 2)`). Один и тот же человек может идти под
  несколькими голосами (в образце у Андрея Потапова есть «голос 1» и «голос 2»).
- **Реплика:** строка вида `[HH:MM:SS] текст`. Тайм-штамп в квадратных скобках, пробел, текст.
- Пустые строки между репликами/блоками.

**Вывод для парсера:** нужен лёгкий строчный конечный автомат с состояниями
`header / speaker / utterance`, а не универсальный JSON/SRT-парсер. Тайм-штампы дают возможность
строить «карту» реплик, а спикеры — фильтры в UI (показывать/прятать реплики конкретного голоса).

### 8.2. Дефекты распознавания (что подтверждает ценность каждого словаря)
Образец богат примерами — буквально под каждое правило из YAML:

| Тип дефекта | Пример из текста | Куда ляжет в словаре |
|---|---|---|
| Замена имени собственого | «Останавливаю **Самри**» (конец файла) | `replacements`: правило `Самри → Саммари` (уже есть) |
| Искажённое имя/термин | «**джем борд / джемборд**», «**жире**» | `replacements` (правила уже есть: → дашборд, → JIRA) |
| Искажённое имя собственое | «**Швитков**», «**Михаил Шветков**» (вместо Шведков), «**Андрей Маспанов**» (вместо Моспанов) | `replacements` (новые правила) |
| Filler-слова и междометия | «ну», «вот», «в общем-то», «так», «прям», «блин», «ээ» | `filler.yaml → filler_words` |
| Filler-фразы | «по большому счёту», «как бы», «то есть», «в принципе» | `filler.yaml → filler_phrases` |
| Неясные/«мусорные» куски | «**И вот тоже еще полкового. Вейбостроителей прибыла**» | подсветка детектором, ручная разметка |
| Перепутанные термины | «**же с ними» (видимо «же с ними» = «джейсоны»/JSC?), «**Ай-яй-яй и Атата**» | детектор / replacements |
| Англо-русская мешанина | «**скилл стулом» (→ к LLM-скиллом), «**токен задач», «**Оверхед»», «**Дему» (демку) | replacements + детектор |

### 8.3. Важные следствия для архитектуры

1. **«Слово» ≠ токен в понимании движка.** Текст смешанный: русские слова, английские вставки
   (`DeckBody`, `Workflow`, `HTML`, `Markdown`, `DSL`, `CSV`), гибриды («скилл стулом»). Движок
   токенизации должен корректно обрабатывать латиницу, дефисы (`код-ревью`), кавычки и не рвать
   англоязычные термины.
2. **Контекст подсветки.** «Нераспознанные»/подозрительные сегменты часто — это не отдельные слова,
   а **фразы** («полкового. Вейбостроителей прибыла», «Ай-яй-яй и Атата»). Подсветка и контекстное
   меню должны уметь работать как со словом, так и с **произвольным выделением** (диапазоном), и
   отправлять в словарь строку/фразу целиком.
3. **Спикер-контекст для замен.** Искажения имён собственых («Шветков», «Маспанов») зависят от
   контекста (кто говорит / о ком). В replacements уже есть `label` — стоит использовать его в UI
   для группировки «имена/термины/бренды», а саму замену применять глобально.
4. **Тайм-штампы как якоря.** При «добавлении в словарь» и в статистике полезно показывать время
   реплики (`[00:48:40]`), чтобы пользователь быстро находил контекст в записи.
5. **Что НЕ трогать.** Фразы типа «Ай-яй-яй и Атата» — это осмысленная речь (иллюстративный пример),
   а не мусор. Это аргумент за режим ручного подтверждения: подсветка — лишь **предложение**, а не
   авто-действие. Подтверждает необходимость `keep_override` в `filler.yaml`.

### 8.4. Уточнение к MVP (этап 2)
Движок правил должен как минимум:
- парсить плоский формат (спикер + `[HH:MM:SS]` + текст);
- применять `filler_words`, `filler_phrases`, `replacements` (с учётом регистрочувствительности
  для имён: «Самри» в начале предложения с заглавной);
- детектить слова вне `detector_whitelist` и подсвечивать их, сохраняя якоря (номер строки,
  тайм-штамп, спикер) для перехода и контекстного меню.

### 9.3. Контракт имён файлов (на основе реальных образцов)

Зафиксируем распознаваемые автодетектом имена и схему каждого:

| Файл (реальный) | Роль | Действие из ПКМ-меню |
|---|---|---|
| `settings.yaml` | глобальные настройки | редактирование (форма) |
| `filler.yaml` | удаление filler/междометий | «Удалить всегда (filler)» / «Не трогать (keep)» |
| `replacements.yaml` | замены + lemma-замены | «Заменить на…» / «Сохранить лемму» |
| `lemma_irregular.yaml` | Flexия→лемма | «Указать лемму» |
| `detector_whitelist.yaml` | белый список детектора | «Добавить в whitelist» |

`custom.yaml` из предложения не нужен — его роль покрывают `settings.yaml` + `keep_override`.

## 10. Предлагаемые форматы YAML

Цель — зафиксировать схемы так, чтобы автодетект, валидация (JSON Schema) и AST-правка работали
стабильно. **Принцип:** минимально менять существующие файлы, усиливая их комментариями-контрактом и
JSON Schema. Формат `replacements.yaml` оставляю **без изменений** (как просил автор — он устраивает).

### 10.1. `replacements.yaml` — БЕЗ ИЗМЕНЕНИЙ ✅

Текущий формат сохраняем как канонический:

```yaml
# Замены строковых фрагментов и лемматизационные замены.
# replacements: точное (строковое) совпадение from -> to.
# lemma_replacements: замена по лемме слова (from_lemmas -> to).
# Ключи replacement_rule_NNN / lemma_rule_NNN — уникальные, нумерация сквозная.
replacements:
  replacement_rule_001:
    to: Minimax
    label: LLM vendor          # необязательно, но полезно для группировки в UI
    from:
    - мини Макс
    - минимакс
    - mini max
lemma_replacements:
  lemma_rule_001:
    to: Slack
    from_lemmas:
    - слак
```

Договорённости для UI/AST-правки:
- Новый `replacement_rule_{NNN}` генерируется как `max(существующие)+1`, формат `%03d`.
- `from` — всегда список (даже из одного элемента), чтобы не ломать структуру.
- Регистр `from` — регистронезависимое сравнение для русско-латинских форм («mini max»), но `to`
  сохраняется как задано (бренд «Minimax»). Для имён в начале предложения — опция
  «заменять с заглавной» (сводится к capitalize после подстановки).

### 10.2. `filler.yaml` — формализуем существующий

```yaml
# Слова/фразы-заполнители и междометия, удаляемые при очистке.
filler_words:        # одиночные слова
- ага
- блин
- короче
filler_phrases:      # целые фразы (многословные обороты)
- как бы то ни было
- по большому счёту
keep_override: []    # исключения: эти слова/фразы НЕ удалять, даже если совпали
```

Замечания: при добавлении из ПКМ «Удалить всегда» кладём в `filler_words`; «Не трогать» →
`keep_override`. По возможности сохранять алфавитный порядок для отображения (без перезаписи файла,
только в UI-сортировке).

### 10.3. `detector_whitelist.yaml` — список + необязательная группировка

Минимально — как сейчас:

```yaml
common_words:
- codereview
- апи
- бэкенд
```

**Опциональное улучшение (без потери совместимости)** — разрешить расширенную форму для новых записей,
чтобы хранить контекст/дату добавления; старый простой вид валиден параллельно:

```yaml
common_words:
- codereview
- апи
- бэкенд
entries:            # необязательная секция; если есть — мерджится с common_words
- value: пайплайн
  note: добавлено из UI
  added: 2026-07-09
```

Если пайплайн-даунстрим читает только `common_words` (плоский список строк) — секцию `entries`
придётся «разворачивать» в `common_words` при сохранении. Проще: **оставить плоский список** и
хранить метаданные в отдельном `.meta.json` рядом (приложение-only), не трогая YAML. Рекомендую второй
вариант, чтобы не плодить форматов.

### 10.4. `lemma_irregular.yaml` — без изменений

```yaml
lemma_irregular:
  люди: человек
  шли: идти
  детей: ребёнок
```

Простой `dict`. Добавление через ПКМ «Указать лемму» — новая пара ключ/значение. Сортировка по
ключу — только в отображении.

### 10.5. `settings.yaml` — расширить с описаниями

```yaml
settings:
  aggressive_clean: false        # агрессивный режим: удалять и «пограничные» filler
  similarity_threshold: 0.65     # порог нечёткого сравнения для подсказок замены (0..1)
  min_word_len: 3                # слова короче этого — кандидаты в «мусор»
  suggestions_header: true       # печатать шапку блока подсказок в выводе пайплайна
  # --- предлагаемое дополнение (необязательно, дефолты указаны) ---
  case_sensitive_names: true     # с учётом регистра для имён собственных в replacements
  keep_timestamps: true          # сохранять [HH:MM:SS] в очищенном выводе
  strip_speaker_meta: false      # убирать «(голос N)» из имени спикера
```

### 10.6. Файл настроек приложения (НЕ в репозиторий)

`transcript-optimizer.config.json` (или через Tauri store) — без секрета:

```json
{
  "recent_dirs": ["C:/.../transcript_optimizer"],
  "last_dir": "C:/.../transcript_optimizer",
  "llm": {
    "base_url": "https://api.openai.com/v1",
    "model": "gpt-4o-mini",
    "temperature": 0.3,
    "max_tokens": 4096
  },
  "ui": { "theme": "dark", "panel_ratio": 0.5 }
}
```

`api_key` — **только** в Windows Credential Manager через `keytar`/Tauri keychain plugin.

## 11. Резюме

Идея здравая и хорошо ложится на современные open-source компоненты. Предложенный стек
(Tauri 2 + React/TS + Monaco + react-markdown/mermaid + Zustand) принимается с точечными
корректировками: `yaml` вместо `js-yaml` (ради CST/комментариев), `react-resizable-panels` вместо
`react-split-pane`, прямой `fetch` к OpenAI-совместимому API вместо SDK, хранение ключа в keychain.

Форматы YAML фиксируем на основе **реальных образцов** (а не выдуманных имён), сохраняя
`replacements.yaml` без изменений по просьбе автора. Ключевые технические задачи —
**AST-безопасная правка словарей** и **интерактивный движок правил** с подсветкой по словам и фразам.
Главный блокирующий вопрос для старта — п. 6.1 (где живёт логика очистки); формат транскрипта уже
изучен (раздел 8).
