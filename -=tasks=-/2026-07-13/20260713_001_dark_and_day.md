## 1. Постановка задачи

Сейчас приложение жёстко оформлено в тёмной теме (по сути — палитра VS Code Dark).
Все цвета захардкожены в `app/src/App.css`, а Monaco-редактор в
`app/src/components/YamlEditor.tsx` захардкожен в `theme="vs-dark"`.

Нужно добавить **светлую тему** и **переключатель тем** с сохранением выбора и
корректной сменой темы Monaco. Поведение:

- Две темы: **Тёмная** (текущая, по умолчанию) и **Светлая**.
- Переключатель в тулбаре (виден в обоих режимах: «Словари» и «Транскрипт»).
- Выбор сохраняется между запусками (localStorage).
- Тема применяется одновременно к UI (CSS) и Monaco-редактору.
- Вариант «Системная» (по `prefers-color-scheme`) — **не входит** в эту задачу
  (см. раздел 9, будущие улучшения). Достаточно явного переключателя.

## 3. Архитектура решения

### 3.1. CSS-переменные как единственный источник цвета

(тёмная — по умолчанию) и переопределяемые в `[data-theme="light"]`:

  --bg-elevated:   #2d2d30;   /* кнопки, поля ввода */

### 3.2. Theme-store (Zustand)

Новый файл `app/src/store/theme.ts`:

```ts
import { create } from "zustand";

export type ThemeMode = "dark" | "light";

interface ThemeState {
  mode: ThemeMode;
  setMode: (m: ThemeMode) => void;
  toggle: () => void;
}

const STORAGE_KEY = "transcript-optimizer.theme";

function initialMode(): ThemeMode {
  try {
    const saved = localStorage.getItem(STORAGE_KEY);
    if (saved === "dark" || saved === "light") return saved;
  } catch { /* localStorage может быть недоступен */ }
  return "dark"; // значение по умолчанию
}

function applyToDom(mode: ThemeMode) {
  document.documentElement.setAttribute("data-theme", mode);
}

export const useTheme = create<ThemeState>((set, get) => ({
  mode: initialMode(),
  setMode: (m) => {
    applyToDom(m);
    try { localStorage.setItem(STORAGE_KEY, m); } catch { /* noop */ }
    set({ mode: m });
  },
  toggle: () => get().setMode(get().mode === "dark" ? "light" : "dark"),
}));

// Применить тему к DOM при первом импорте store (чтобы CSS-переменные
// совпали с mode ещё до первого рендера компонентов, подписанных на store).
applyToDom(useTheme.getState().mode);
```

Почему store, а не локальный `useState` в App:
- Monaco-редактор в `YamlEditor` должен реагировать на смену темы — ему нужен
  внешний источник правды, доступный по импорту.
- Toolbar рисует кнопку; App/TranscriptView/etc. читают только CSS-переменные,
  но единый store делает механизм тривиально расширяемым (п. 9).

### 3.3. Раннее применение темы (анти-FOUC)

Если тело страницы успеет отрисоваться с `:root` (тёмная) до того, как React
прочитает localStorage, будет вспышка тёмной темы при выборе светлой. Решение —
маленький инлайн-скрипт в `index.html` перед `main.tsx`:

```html
<script>
  (function () {
    try {
      var t = localStorage.getItem("transcript-optimizer.theme");
      if (t === "light" || t === "dark") {
        document.documentElement.setAttribute("data-theme", t);
      }
    } catch (e) { /* noop */ }
  })();
</script>
```

Ключ `STORAGE_KEY` в `theme.ts` должен дословно совпадать с этим литералом.
CSP в `tauri.conf.json` сейчас `null` — инлайн-скрипт разрешён; менять не нужно.

### 3.4. Интеграция Monaco

`@monaco-editor/react` отслеживает проп `theme` и вызывает
`monaco.editor.setTheme(...)` при его изменении. Поэтому достаточно
переподписать редактор на `useTheme`:

```tsx
// YamlEditor.tsx
import { useTheme } from "../store/theme";
// ...
const mode = useTheme((s) => s.mode);
// ...
<Editor
  ...
  theme={mode === "dark" ? "vs-dark" : "light"}
  ...
/>
```

Дополнительно при первом монтировании тема уже применена (см. onMount),
но `@monaco-editor/react` самообрабатывает смену — ручной вызов
`monaco.editor.setTheme` **не требуется**. Это проверяется в п. 8.

### 3.5. Переключатель в тулбаре

В `Toolbar.tsx` переключатель должен быть виден в обоих режимах
(«Словари» и «Транскрипт»). Самый простой вариант — иконка-переключатель,
помещённая в начале JSX обоих `return` (до кнопок режима), чтобы он не
уезжал в `status` (у того `margin-left: auto`):

```tsx
// общий кусок для обоих return:
<button
  className="btn theme-toggle"
  onClick={() => toggle()}
  title={mode === "dark" ? "Переключить на светлую тему" : "Переключить на тёмную тему"}
  aria-label="Переключить тему"
>
  {mode === "dark" ? "☼" : "☾"}
</button>
```

Символы `☼`/`☾` — без зависимости от иконочного пакета. При желании позже
заменить на SVG (п. 9). Подписка:

```tsx
const mode = useTheme((s) => s.mode);
const toggle = useTheme((s) => s.toggle);
```

- [ ] Переключение мгновенно меняет UI **и** тему Monaco (без перезагрузки).
- [ ] Выбранная тема сохраняется между запусками (проверить: закрыть/открыть окно).
- [ ] Нет FOUC тёмной темы при сохранённой светлой (проверить запуск).
- [ ] Все элементы читаемы в светлой теме: тулбар, вкладки, edit-panel (включая
      поля ввода/ошибки), mode-switch, транскрипт-панели, легенда, статистика,
      diff-блок (`edit-diff`), Monaco.
- [ ] Подсветка подозрительных слов в транскрипте (`legend .dot.oov|will-replace|filler`)
      остаётся различимой в обеих темах.
- [ ] `pnpm exec tsc --noEmit` и `pnpm exec vite build` проходят без ошибок.
- [ ] Закрытие окна с несохранёнными правками по-прежнему спрашивает подтверждение

## 6. Риски и нюансы

- **Забытые литералы.** В `App.css` ~30 правил с цветом; легко пропустить
  (особенно `border-bottom`, `rgba(...)` в `.legend .dot.*`, `!important` в
  `.stats-stale`). После перевода — пройти `grep -n "#[0-9a-fA-F]" app/src/App.css`
  и убедиться, что остались только значения внутри переменных.
- **`!important`.** `.stats-stale` использует `color: #d4a85a !important;` — при
  переводе на `var(--warn) !important;` сохранить `!important`, иначе потеряет
  приоритет у `b`/`span`.
- **Monaco при старте.** Если Monaco смонтировался до применения темы — будет
  короткая вспышка. Инлайн-скрипт в `index.html` + `applyToDom` на уровне модуля
  store это покрывают; проверить визуально.
- **Zustand StrictMode двойной mount.** Theme-store без побочных эффектов в
  конструкторе (applyToDom вызывается один раз на уровне модуля) — регрессии нет.
- **RegEx в селекторах.** В `App.css` есть `.diff-\+` / `diff\-` (escapes в CSS) —
  трогать только цвет, не селекторы.
- **Tauri-окно.** Заголовок/рамка окна рисуются ОС и не подчиняются CSS-теме —
  это ожидаемо и за рамками задачи.

## 7. Цветовая палитра (рабочие значения)

Отправные значения см. в п. 3.1. Светлая палитра подобрана «по духу» VS Code Light
+ читаемый контраст. Во время шага 7 сверить фактический вид и при необходимости
скорректировать: в первую очередь `--bg-surface`/`--bg-input` (должны быть
различимы), `--text-secondary` (контраст ≥ 4.5:1 к фону), `--accent`
(достаточно насыщенный для активной вкладки).

Палитра Monaco: встроенные `"vs-dark"` и `"light"` — **без кастомных тем**.
Если позже понадобится единый акцент с UI — `monaco.editor.defineTheme(...)`
(вне рамок этой задачи, см. п. 9).

## 8. Проверка

После реализации:

```bash
cd app
pnpm exec tsc --noEmit      # типы
pnpm exec vite build        # сборка фронтенда
pnpm tauri dev              # ручной прогон
```

Ручной чек-лист:
1. Запуск с чистым localStorage → тёмная тема (по умолчанию).
2. Нажать переключатель → светлая; UI и Monaco поменялись одновременно.
3. Перезапустить → сохранилась выбранная тема.
4. Пройти по обоим режимам («Словари» и «Транскрипт») в обеих темах.
5. Открыть транскрипт, нажать «Очистить» → проверить читаемость diff и легенды.
6. Открыть папку словарей → Monaco, edit-panel, ошибка валидации (`.edit-err`)
   читаемы в обеих темах.
