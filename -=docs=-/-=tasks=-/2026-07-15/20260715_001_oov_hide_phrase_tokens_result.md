## Что реализовано (факты)

### Постановка
Панель «Статистика OOV» показывала фантомные OOV: токены, которые пользователь
уже обработал через многословное правило `filler_phrases` или `replacements[*].from`,
но движок их не исключал из OOV (его проверки `replaceIdx.has`/`fillerWords.has`
работают только для одиночных значений).

**Решение (принято пользователем 2026-07-15):** скрывать токен из грида, если
**все** его вхождения лежат внутри какой-либо многословной фразы (≥ 2 слов после
нормализации) из `filler_phrases` или `replacements[*].from`. Если хотя бы одно
вхождение — снаружи фразы, токен остаётся с полным `count` (внутри и снаружи).

### Арена изменений
- **Движок (`rules.ts`) — НЕ ТРОНУТ.** Источник истины decorations не меняется;
  фильтр — это статический анализ текста+правил, он естественно лежит на
  агрегаторе грида, а не на движке (урок — см. `LESSONS_LEARNED.md` §6).
- **`OovStatsGrid.tsx` — НЕ ТРОНУТ.** Контракт `OovRow` не изменился, просто
  строк может стать меньше.
- **`app/src/engine/oov-stats.ts`** — расширена сигнатура `buildOovRows`,
  добавлены приватные утилиты и новый тип `OovStatsContext`.
- **`app/src/components/TranscriptView.tsx`** — `useMemo` для контекста,
  проброс `transcript.parsed` и контекста в `buildOovRows`.

### `app/src/engine/oov-stats.ts`
- Новый экспорт:
  ```ts
  export interface OovStatsContext {
    phraseNorms: string[]; // нормализованные фразы (≥ 2 слов)
  }
  ```
- Сигнатура:
  ```ts
  buildOovRows(
    cleanResult: CleanResult,
    transcript?: ParsedTranscript | null,
    ctx?: OovStatsContext,
  ): OovRow[]
  ```
  Старый одноаргументный вызов остаётся совместимым (операнды 2/3 опциональны).
- Приватные helper'ы (новые):
  - `collectPhraseSpans(transcript, phraseNorms)` → `Map<lineNo, PhraseSpan[]>`
    (по `utt.text`, без префикса `[HH:MM:SS]`).
  - `collectTokenOccurrences(transcript)` → `Map<norm, {lineNo,start,end}[]>`
    (тот же `tokenize`, что и движок — нормализация и границы слов совпадают).
  - `wordBoundaryRe` + `escapeRe` — локальные копии логики из `rules.ts`,
    чтобы не экспортировать движковые приватные helper'ы. Те же Unicode property
    escapes `\p{L}\p{N}`, флаг `u`, регистронезависимо.
- **Алгоритм фильтрации:** для каждого `key ∈ map.keys()` собираются все его
  вхождения в транскрипте; если **каждое** вхождение `⊂` хотя бы одного
  `phraseSpan` на его `lineNo` — `key` удаляется из map.
- **Сохранена сортировка:** по `count` ↓, затем по `display` ↑ (стабильно).

### `app/src/components/TranscriptView.tsx`
- Импорт расширен: `import { buildOovRows, type OovStatsContext } from "../engine/oov-stats"`.
- Новый `useMemo<OovStatsContext>` — собирает `phraseNorms` из:
  ```ts
  filler?.filler_phrases ?? []   // .map(norm).filter(n => n.includes(" "))
  Object.values(replacements?.replacements ?? {}).flatMap(rule =>
    (rule.from ?? []).map(norm).filter(n => n.includes(" "))
  )
  ```
  Зависимости `[filler, replacements]` — мемоизация пересчитывается только при
  изменении словарей, не при перерисовке.
- `useMemo` для `oovRows` теперь принимает `transcript.parsed` и `oovCtx` —
  пересчёт триггерится при изменении транскрипта, словарей или результата
  очистки.

## Lessons Learned (дубль §6)

### 5. Где фильтровать — в агрегаторе, а не в движке
Движок `rules.ts` остался без изменений. Источник истины для decorations
не размыт «смежной» UI-заботой. Это позволило:
- не плодить новых краевых эффектов в `applyRules` (unit-style мыслимость);
- снять фразовую фильтрацию как чистую функцию, мемоизируемую через `useMemo`.

### 6. Те же `tokenize`/`norm`, та же `wordBoundaryRe` — на кириллице разъедется
Если в агрегаторе использовать «свой» токенизатор или другую логику word-boundary,
позиции токенов не совпадут с движковыми → фраза «в общем» может матчиться,
а её токен «общем» — нет (или наоборот). Решение — импортировать
`tokenize`/`norm` из `engine/tokenizer.ts`, а для границ слов — копировать
логику `wordBoundaryRe` движка (Unicode property escapes, флаг `u`).

### 7. По `utt.text`, не по `transcript.raw`
В исходном `transcript.raw` есть префикс `[HH:MM:SS]` (длина 11), который
ломает колоночные координации. Движок работает по `utt.text` (без префикса,
после парсера) — поэтому и наш второй проход идёт по `utt.text`.
`tokenize(utt.text)` возвращает координаты `[start,end)` ровно в той же
системе отсчёта, что и `decoration.text` (см. `rules.ts:178`, `origText = utt.text`).

## Сценарий ручной проверки

(для следующего, кто будет смотреть)

1. Открыть транскрипт → «Очистить» → вкладка «Статистика (N)».
2. Запомнить N (число строк грида).
3. В `filler.yaml` или `replacements.yaml` добавить многословное правило,
   содержащее слово, которое **до правки** было в гриде.
4. Сохранить словарь → грид загорелся `⚠ устарел` → «Очистить».
5. Убедиться, что это слово **исчезло** из грида (N уменьшилось).
6. Добавить то же слово **где-то вне** многословной фразы (например, как
   одиночный `from` в другом правиле, или просто произнести его в реплике).
7. «Очистить» — слово должно **вернуться** в грид с общим `count`.
8. Кнопки «Добавить в whitelist» / «Добавить в replacement» по оставшимся
   словам работают как раньше.
