# Дедупликация `replacements.yaml` по каноническому `to` + показ `to` в списке правил

**Дата:** 2026-07-15
**Проект:** transcript-optimizer (desktop: Tauri 2 + React 19 + TS + Vite 7 + Monaco)
**Ветка:** `docs/transcript-optimizer-plan` (продолжаем)
**Этот файл — план следующего изменения.** Реализуется по нему.

---

## ❗ Проблема

В `sample/transcript_optimizer/replacements.yaml` накопились правила, продублированные по канонической форме (`to`). Дубликаты лежат как валидно оформленные, но семантически одно и то же — движок для каждого `from` сработает, но глоссарий/UI/OOV-статистика показывают несколько «правил про одно и то же».

**Реальный пример из текущего словаря (подтверждено `replacements.yaml` после ручной правки пользователем):**

```yaml
replacement_rule_037:
  to: Python
  label: tech_term
  from: [питанящих]
replacement_rule_043:
  to: Python
  label: tech_term
  from: [питоновском]
```

```yaml
replacement_rule_049:
  to: плохо
  label: linguistic
  from: [Атата]
replacement_rule_050:
  to: плохо
  label: linguistic
  from: [пипец-пипец]
```

**Что плохо:**

- В OOV-гриде появляются «ложные» упоминания Python там, где одно правило — это уже синонимы Python. Несколько правил с одним `to` усложняют чтение словаря.
- В глоссарий экспортируется несколько определений на одну сущность (или ни одного — `description` есть только у одного из пары, но без дедупа это не помогает).
- В правой панели (`EditPanel`) сейчас выводится только `rule_key` (`replacement_rule_015`) — приходится открывать YAML, чтобы вспомнить, какое это слово. На длинном словаре это боль.

**Что не плохо:**

- Текущая валидация `validate.ts::replacements:to-duplicate` уже **ловит** это — но как ошибку, без кнопки «исправить». Пользователь ловит маркер Monaco после правки, идёт чинить руками. Хотим: дедупликация по кнопке.

---

## 🎯 Цель

1. **Кнопка «Дедуплицировать по to»** в `EditPanel` (только когда активная вкладка — `replacements`). Показывает превью-diff (что будет смержено), требует подтверждения через «Применить» — не применяется сразу.
2. **В списке правил** (`edit-list`) **рядом с ключом** выводить канонический `to` в круглых скобках: `replacement_rule_015 (n8n)`.

### Критерии готовности

1. Точные дубликаты `from` (как в прошлой фиче) — то же текущее поведение, ничего не сломано.
2. Кнопка «Дедуплицировать» в `EditPanel` доступна только при `activeKind === "replacements"`.
3. **Сравнение `to` идёт по нижнему регистру (lowercase) + trim.** Это значит:
   - `Python` и `python` — одно и то же правило;
   - `Python` и ` Python ` — одно и то же правило;
   - `Excel` и `Excelских` — **разные** правила (не нормализуем «основу слова», только регистр и пробелы).
4. Клик на «Дедуплицировать»:
   - вычисляет новый raw без дублей по `to` в `replacements.*`,
   - кладёт результат в `pending`/`pendingBefore` (тот же механизм, что для превью при добавлении),
   - в diff видно: правила с дублями `to` сливаются в первое по ключу, остальные удаляются.
5. «Применить» фиксирует изменения, «Отмена» сбрасывает preview.
6. После применения кнопка снова доступна; повторное нажатие не меняет ничего (raw уже без дублей) — покажется сообщение «Дублей по to нет — словарь чист».
7. Список правил `edit-list` для `replacements` показывает ключи в формате `replacement_rule_015 (n8n)`. Для lemma — `lemma_rule_010 (Slack)`. Для glossary — без изменений (`id`, категория читаема).
8. Удаление правила из списка (кнопка `✕`) работает как раньше — на полном ключе `replacement_rule_NNN`, суффикс `(to)` только визуальный.
9. **Покрытие — только `replacements`** (то, что пользователь указал). `lemma_replacements` пока не трогаем — там дублей нет, и формат задачи — отдельный кейс.
10. `pnpm exec tsc --noEmit` — чисто.
11. `pnpm exec vite build` — `✓ built`.
12. **Сценарий пользователя:** имеем два правила `_037`/`_043` → `Python`/`python` (любой регистр). Нажимаем «Дедуплицировать» → превью показывает:
    ```
    - replacement_rule_037:
    -   to: Python
    -   label: tech_term
    -   from:
    -   - питанящих
    - replacement_rule_043:
    -   to: python
    -   label: tech_term
    -   from:
    -   - питоновском
    + replacement_rule_037:
    +   to: Python
    +   label: tech_term
    +   from:
    +   - питанящих
    +   - питоновском
    ```
    «Применить» → готово. Список правил показывает одно `_037 (Python)`.

### ВНЕ MVP (явно)

- Дедупликация по `to` в `lemma_replacements`. Там формат другой (`from_lemmas` → `to`), и нормализация иная.
- Дедупликация по «очень похожему `to`» (`Codex` vs `кодексе` — это разные слова, не дубль; определяется вручную). Тоже отдельная фича.
- Автоматический пуск при «Применить» формы (когда пользователь пытается создать новое правило `to=Python` при существующем — просить объединить). Это похоже на логику панели «Похожие from», но для `to`. Осознанно не делаем: пользовательский запрос именно про ручную разовую кнопку.
- Mass-merge нескольких `to` в один через диалог с выбором, какое `to` оставить.
- Фонематическое/стемматическое сравнение (`Excel` vs `Excelских`). Это другая задача — отдельная фича.

## 🧠 Решение

### 1. Алгоритм дедупликации (`yaml-edit.ts`)

**Новая публичная функция `dedupByTo(raw): EditResult & { noop?: boolean }`**, расширяем `EditResult` опциональным `noop`. Алгоритм:

```
dedupByTo(raw):
  1. parseDocument(raw) — если ошибки, вернуть ok:false
  2. для секции `replacements`:
       - обойти все правила по ключам в исходном порядке;
       - нормализовать to: norm(to).trim() — LOWERCASE + пробелы;
         "Python" → "python", " python " → "python".
       - построить Map<norm(to), { firstKey, others[] }>;
       - если ВСЕ группы размера 1 (нет дублей) — вернуть { raw, ok: true, noop: true };
       - для каждой группы с размером > 1:
            * взять first = первое правило (по ключу — самый маленький номер);
            * собрать уникальные from из ВСЕХ правил группы:
                - Set<string> по нормализованному from (norm + схлопнуть пробелы);
                - написание ВЫБИРАЕМ первое встреченное (стабильно для diff);
            * first.from[...] = объединённый уникальный список;
            * удалить остальные правила группы через doc.deleteIn(["replacements", key]).
  3. вернуть новый raw через toString(SERIALIZATION_OPTS)
```

**Ключевые детали:**

- **Нормализация `to` для сравнения:** `norm(raw).trim()` через `engine/tokenizer.ts` (`norm = s.toLowerCase()`). **Регистр не различает, пробелы по краям — не различает, остальное — различает.** Это инвариант зафиксирован пользователем 2026-07-15: «сравнивай приведённые к нижнему регистру».

- **Уникальность `from` при мерже:** `Set<string>` ключей по нормализованному значению (`norm(from).replace(/\s+/g, ' ').trim()`), плюс сохраняем **первое встреченное** написание для каждого ключа. Так `Excel` и `excel` не задвоятся, и не потеряется ни одно из исходных написаний.

- **Порядок `from[]` в результате:** правила группы перебираются в порядке ключей (от `_037` к `_043`), внутри правила — как было. Логика: «первый встреченный побеждает», не алфавитная сортировка.

- **Удаление правил через CST:** `doc.deleteIn(["replacements", ruleKey])`. Комментарии и стиль сохраняются.

- **`label` берётся от первого правила** (по ключу — он старейший). Не мержим `description` — если у других правил было `description`, оно тихо отбрасывается (видно в diff как минус-строки).

- **`noop`:** если ВСЕ группы дублей имеют размер 1 (то есть ни одного дубля), возвращаем `{ raw, ok: true, noop: true }`. `EditPanel` покажет сообщение «Дублей по to нет — словарь чист» и НЕ поставит pending-diff.

- **Пустые `to`.** Если правило без `to` (нормализация даст пустую строку) — **не участвует в группах** (защита от случайного слияния битых правил). Правила с пустым `to` остаются как есть.

- **Отображение `to` в списке правил (UI):** показываем КАНОНИЧЕСКИЙ текст из YAML (с реальным регистром/пробелами), не нормализованный. То есть если в словаре `to: Python` — в списке будет `(Python)`, даже если в другой записи было `to: python`. Это потому что в результате дедупа **сохраняется написание первого правила** (через CST-`firstKey`), и пользователь видит то, что в итоговом словаре.

### 2. UI в `EditPanel.tsx`

**(a) Кнопка «Дедуплицировать по to»:**

- Размещение: в `edit-panel-header`, рядом с существующим «↶ Undo».
- Видна только при `activeKind === "replacements"`.
- Disabled при `pending !== null` (пока открыт другой preview).
- По клику:
  ```ts
  function handleDedup() {
    setErr("");
    if (!activeEntry) return;
    const res = dedupByTo(activeEntry.raw);
    if (!res.ok) { setErr(res.error ?? "ошибка"); return; }
    if (res.noop) {
      setErr("Дублей по to нет — словарь чист");
      setPending(null);
      return;
    }
    setPendingBefore(activeEntry.raw);
    setPending(res.raw);
  }
  ```
- Тот же `pending`/`setPending`/`pendingBefore`, что для добавления нового правила. Кнопки «Превью»/«Применить»/«Отмена» работают без изменений — пользователь видит diff и применяет.

**(b) Список правил `edit-list`:**

- Тип меняется с `string[]` на `Array<{ key: string; to?: string }>` (только для `replacements`/`lemma_replacements`). Для `glossary` оставляем `string[]` (там нет `to`).
- Разметка:
  ```tsx
  <li>
    <span>{k.key}{k.to ? <> (<code>{k.to}</code>)</> : null}</span>
    <button onClick={() => handleDelete(k.key)}>✕</button>
  </li>
  ```
- `handleDelete` уже получает «голый» ключ — не зависит от того, как список отображается.

### 3. Что НЕ трогаем

- `ContextActionDialog.tsx` — без изменений.
- `lib/validate.ts` — без изменений. Точные дубликаты (`from-conflict`, `to-duplicate`) по-прежнему ловятся и подсвечиваются в Monaco.
- `engine/*` — без изменений.
- `types/*` — без изменений.
- `store/*` — без изменений.

### 4. Архитектурное правило: «операции над raw → в `yaml-edit.ts`»

Следуем той же логике, что и `addEntry`/`appendFromToRule`/`findRuleLine`: дедупликация — чистая функция над raw через CST, тестируемая в изоляции. `EditPanel` — только UI-обёртка.

## 🔧 План реализации (по шагам)

### Шаг 1. `app/src/lib/yaml-edit.ts` — новая функция `dedupByTo`

- Расширить `EditResult` опциональным `noop?: boolean` (либо ввести `DedupResult extends EditResult`).
- Реализовать `dedupByTo(raw: string): { raw, ok, error?, noop? }`:
  - `parseDocument(raw)` → проверка `doc.errors.length === 0`,
  - получить секцию `replacements` через `doc.get("replacements", true)`,
  - обойти пары в исходном порядке через `block.items`,
  - для каждой записи: `toValue = String(pair.value.to?.value ?? '').trim()`; `toKey = tokenNorm(toValue)`; если `toKey === ''` — пропустить (битое правило);

  - для каждой группы с `others.length > 0`: расширить `from` первого правила, удалить остальные через `doc.deleteIn(["replacements", key])`,
  - вернуть новый raw через `doc.toString(SERIALIZATION_OPTS)`.
- Уделить внимание:
  - правило без `to` (пустое/битое) — пропускаем, оно не группа.
  - правило без `from` / `from` не список — `fromNode.items` пустой; мерж ОК.
  - дубль с одинаковым `from` — `Set` отсечёт дубль, ничего не сломается.

## ⚠️ Риски и нюансы

1. **Раздел «Дубли» в Monaco-markers.** Сейчас `validate.ts::replacements:to-duplicate` показывает каждый дубликат как error-маркер. После дедупа — маркеры исчезнут (правильно). Если пользователь НЕ применил превью, маркеры остаются.
2. **`description` отбрасывается.** Если у `_037` было `description: "Язык программирования Python"`, а у `_043` не было — после мержа сохранится только то, что у первого правила (по ключу). Это минимально инвазивно и обратимо через undo.
3. **Кросс-секции.** `replacements` и `lemma_replacements` имеют разные `to`-пространства. Дедуп НЕ мержит кросс-секционно — только внутри `replacements`. Логично оставить так (разная семантика).
4. **Пустые `to`.** Защита: правила с пустым `to` (после `norm().trim()`) пропускаются и не мерджатся — иначе можно случайно слить кучу битых правил в одну группу.
5. **`label` отбрасывается / перетирается.** У всех правил в группе берётся `label` от первого по ключу. Это видно в diff.
6. **`from` у нормализованной формы vs у display-формы.** `Set` строится по нормализованной форме (lowercase + пробелы), но в итоговый `from[]` добавляется то написание, что **первое встречено**. Так `codex` не будет задвоен, если есть `Codex` и `codex`. Семантически правильно: движок матчит по нормализованному.
7. **Регистр `to` в UI.** В списке правил выводим то написание, что в YAML (не нормализованное). Сохраняем «как видим в файле». Если в исходнике было `to: python` (первое по ключу правило — lowercase), но в YAML уже лежит `to: Python` от более позднего правила — выводим то, что осталось после CST-операций (первое по ключу).
8. **Размер diff при большом словаре.** Если у пользователя 5 групп по 3 правила — diff будет длинный. Существующий `computeDiff` уже обрезает до 60 строк (`.slice(0, 60)`). Приемлемо для MVP.
