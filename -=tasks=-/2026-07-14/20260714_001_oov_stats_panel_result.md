## Что реализовано (факты)

### Постановка
Левая панель оригинала была одним Monaco-редактором. Требовалось превратить её
в панель с двумя вкладками: «Оригинал» (как было) и «Статистика» — грид слов,
не найденных ни в одном словаре и в replacement, с множественным селектом и
кнопками добавления в `detector_whitelist.yaml` и `replacements.yaml`.

### Движок — расширение `Decoration`
- `app/src/engine/types.ts`: `Decoration.text?: string` — исходное написание
  токена (как в тексте, без нормализации). Заполняется только для oov/short-garbage.
- `app/src/engine/rules.ts`: `addDecoration` принимает опциональный `text`;
  oov/short-garbage декорации кладут `tok.value`. Остальные категории (filler,
  will-replace) `text` не передают.
- **Зачем:** панель статистики получает слово прямо из декорации, без хрупкого
  обратного поиска по `lineNo`/`startCol`/`endCol` в сыром `raw` (где CRLF и
  префикс `[HH:MM:SS] ` могли бы сбить колонки). Данные уже есть в `tok.value`.

### Агрегация OOV
- `app/src/engine/oov-stats.ts` (**новый**): `buildOovRows(cleanResult): OovRow[]`.
  - Фильтрует `decorations` по `category ∈ { "oov", "short-garbage" }`.
  - Группирует по нормализованной форме (`norm()` = `toLowerCase()`), считая
    частоту; отображаемое значение — первое встретившееся исходное написание.
  - Сортировка: по `count` убыванию, затем `display` по алфавиту (стабильно).
- Тип `OovRow { display, norm, count, category }`.
- Источник данных — уже вычисленные `decorations`, отдельной токенизации нет
  (избегаем двойной работы и рассинхрона с подсветкой в Monaco).

### Грид
- `app/src/components/OovStatsGrid.tsx` (**новый**): таблица с колонками
  «чекбокс | слово | кол-во | тип» (бейдж OOV/short), чекбокс «выбрать все» в
  шапке, клик по строке переключает выделение.
- Множественный селект на `Set<string>` (ключ — `norm`), footer со счётчиком
  «Выбрано: N» и двумя кнопками:
  - **«Добавить в whitelist»** — активна при `N ≥ 1`.
  - **«Добавить в replacement»** — активна **только при `N === 1`** (требование ТЗ).
- Сортировка по count ↓ (дополнительная UI-сортировка в бэклог).

### Вкладки в левой панели
- `app/src/components/TranscriptView.tsx`: левый `.transcript-pane` теперь имеет
  полоску вкладок `.pane-tabs` («Оригинал» / «Статистика (N)», где N — число
  уникальных OOV-слов).
- **Monaco не размонтируется** при переключении вкладок — оборачивается в
  `.pane-tab-body` со скрытием через CSS-класс `hidden` (`display: none`).
  Иначе слетают decorations, позиция курсора и скролл.
- `useEffect` на `origTab` вызывает `editor.layout()` через `requestAnimationFrame`
  при возврате на «Оригинал» — Monaco в скрытом контейнере не пересчитывает
  размеры сам (известный глюк).
- Легенда подсветки (OOV/замена/filler) перенесена в полоску вкладок и показывается
  только на вкладке «Оригинал».
- Вкладка «Статистика» до «Очистить» показывает заглушку (decorations считаются
  только в `applyRules`).

### Обработчик «Добавить в whitelist» (батч)
- Накатывает `raw` одного `addEntry` на вход следующего (reduce по выделенным
  словам) → **один `applyEdit("whitelist", finalRaw)` = один undo-шаг**.
  Несколько `applyEdit` подряд дали бы N undo-шагов — неудобно.
- Дедупликация по `norm()` против уже существующих `common_words`: дубли
  пропускаются с подсчётом «уже было: M».
- Данные словаря (`whitelistRaw`, `applyEdit`) — через zustand-селекторы, НЕ
  `getState` в колбэке (см. `LESSONS_LEARNED.md` §3 — иначе устаревшее значение).

### «Добавить в replacement» — переиспользование диалога
- Устанавливает `ctxAction = "replace"`, `ctxSelection = { text: word, isPhrase: false }`
  → открывается **существующий** `ContextActionDialog`. Никакой новой логики:
  та же форма `to` + `label`, превью diff, apply. Дублировать нельзя.

### CSS
- `app/src/App.css`: `.pane-tabs`, `.pane-tab` (+ `.active`), `.pane-tab-body`,
  `.pane-tab-body.hidden`, `.oov-grid-wrap`/`.oov-grid-scroll`, бейджи
  `.oov-badge-oov`/`.oov-badge-short-garbage`, `.oov-footer`. Использованы
  существующие CSS-переменные (`--bg-surface`, `--accent`, `--legend-*`).

### Правки словарей (через UI, ручной тест фичи)
- `sample/transcript_optimizer/detector_whitelist.yaml`: +~1010 слов в
  `common_words` — добавлены батчами из новой вкладки «Статистика» в ходе
  проверки фичи на реальном транскрипте. Подтверждает, что батч-добавление и
  сохранение структуры/комментариев YAML работают корректно.

### Проверки (все зелёные)
- `tsc --noEmit` — чисто (с `noUnusedLocals`/`noUnusedParameters`).
- `vite build` — `✓ built` (предупреждение о размере чанка — ранее
  существовавшее, не связано с правкой).
- Ручной тест: открыть транскрипт → «Очистить» → вкладка «Статистика» →
  множественный селект → «Добавить в whitelist» (появились все слова, одним
  undo-шагом) → выбрать одно → «Добавить в replacement» (открылся диалог,
  превью/применить работают) → возврат на «Оригинал» (Monaco, decorations и
  позиция курсора на месте).

## Что НЕ сделано (вне MVP, осознанно)
- Сортировка/фильтр/поиск по гриду (только сортировка по count ↓ по умолчанию).
- Группировка по спикеру/таймлайну; пагинация/виртуализация (бэклог).
- Добавление в `filler`/`lemma`/`keep` из этой панели (оставлено для ПКМ-меню).
- Автопересчёт статистики после правки словаря: обновляется через существующий
  флоу — флаг `cleanDirty` + повторное «Очистить» (как в прежней панели статистики).

## Lessons Learned

### 1. Monaco в скрытой вкладке: `display:none` + `layout()` при показе
Monaco не пересчитывает геометрию, пока контейнер скрыт (`display:none`, нулевые
размеры). При возврате на вкладку редактор мог оказаться с неверной шириной/высотой
или серым холстом. **Решение:** `useEffect` на смену вкладки вызывает
`editor.layout()` через `requestAnimationFrame` (чтобы отработал показ контейнера).
Альтернатива — `visibility:hidden`/`position:absolute` — держит редактор в layout,
но тяжелее и не нужна. **Урок: для переключения вкладок с Monaco — прятать через
`display:none`, но дёргать `layout()` при активации; не рассчитывать на
`automaticLayout` в скрытом состоянии.**

### 2. Decorations — уже готовый датасет для статистики, отдельная токенизация не нужна
OOV/short-garbage — это ровно то, что нужно для грида «слов не найденных ни в одном
словаре и replacement». Делать отдельный проход по тексту значило бы: (а) двойная
работа; (б) риск рассинхрона с подсветкой Monaco. **Решение:** брать данные из
`cleanResult.decorations`, добавив в декорацию исходное написание токена (`text`).
**Урок: если движок уже вычислил и подсветил сущности — переиспользовать их как
источник данных для UI, не дублировать логику.**

### 3. Батч-правки в один undo-шаг: reduce raw, один `applyEdit`
`yaml-edit.addEntry` правит CST и возвращает новый `raw`. Для N выделенных слов
наивный цикл `applyEdit` дал бы N undo-шагов — откатывать неудобно. **Решение:**
`reduce` по словам, накатывая `raw` одного `addEntry` на вход следующего, затем
**один** `applyEdit` с финальным `raw`. undo откатывает весь батч разом. **Урок:
для массовых AST-правок — накапливать изменения в сыром тексте и применять одним
store-вызовом, чтобы undo был осмысленным.**

### 4. Zustand-данные в колбэках — через селекторы, не `getState`
`handleAddWhitelist` использует `whitelistRaw` и `applyEdit`. Можно было
`useDictionaries.getState()` внутри колбэка, но это ломает зависимость от рендера
и в будущем может дать устаревшее значение после правки. **Решение:** подписаться
на нужные поля через `useDictionaries((s) => …)` и передать в `useCallback` как
deps. Чуть больше подписок, но корректно. Свериться с `LESSONS_LEARNED.md` §3.
**Урок: в событийных колбэках читать store через хуки + deps, не через getState.**
