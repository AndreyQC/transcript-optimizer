## Постановка

В `sample/transcript_optimizer/replacements.yaml` накопились правила, продублированные по канонической форме (`to`). Реальный пример из словаря пользователя:

```yaml
replacement_rule_037:
  to: Python
  label: tech_term
  from: [питанящих]
replacement_rule_043:
  to: Python
  label: tech_term
  from: [питоновском]

replacement_rule_049:
  to: плохо
  label: linguistic
  from: [Атата]
replacement_rule_050:
  to: плохо
  label: linguistic
  from: [пипец-пипец]
```

Точные дубликаты `to` ловились `validate.ts::replacements:to-duplicate`, но как ошибка — без кнопки «исправить». Список правил в `EditPanel` показывал только ключи (`replacement_rule_015`), без `to` — на длинном словаре приходилось лезть в YAML, чтобы вспомнить, какое это слово.

## Решение (по итогам обсуждения 2026-07-15)

1. **Кнопка «⚙ Дедуплицировать»** в `EditPanel` (только когда активна вкладка `replacements`).
2. **Сравнение `to` — lowercase + trim.** Зафиксировано пользователем: «сравнивай приведённые к нижнему регистру».
3. **Слияние правил**: первое по ключу (старейшее) побеждает, его `from[]` расширяется объединением; остальные правила группы удаляются.
4. **В списке правил** рядом с ключом выводится `to` в круглых скобках: `replacement_rule_015 (n8n)`.

## Что реализовано

### Арена изменений (что НЕ тронуто)

- **Движок `rules.ts`** — без изменений. Дедупликация по `to` не затрагивает матчинг `from` (движок и так правильно работает с нормализованными `from`).
- **`lib/validate.ts`** — без изменений. `replacements:to-duplicate` остаётся как «страховка»: маркер Monaco подсвечивает дубли `to`, и после применения дедупа маркеры исчезают.
- **`engine/*`, `types/*`, `store/*`** — без изменений.
- **`ContextActionDialog.tsx`** — без изменений. Панель «Похожие from» работает как раньше.
- **`SimilarFromPanel.tsx`, `similarity.ts`, `YamlEditor.tsx`** — без изменений.

### `app/src/lib/yaml-edit.ts`

- Расширен `EditResult` опциональным `noop?: boolean` (для дедупа — и в перспективе для других «идемпотентных» операций).
- Импортирован `norm as tokenNorm` из `engine/tokenizer.ts` для сравнения `to`.
- Новая публичная функция `dedupByTo(raw: string): EditResult` — через CST, сохраняет комментарии и стиль:
  - `parseDocument(raw)` → ошибки парсинга → `ok: false`,
  - обход пар секции `replacements` в исходном порядке через `block.items`,
  - нормализация `to` для сравнения: `tokenNorm(to).trim()` (lowercase + trim пробелов по краям); пустые `to` пропускаются (битые правила),
  - группировка по нормализованному `to` через `Map`,
  - для каждой группы с размером > 1: собирается объединение уникальных `from` (через `Set` по нормализованному значению, написание первого встреченного сохраняется), `firstNode.set("from", mergedFrom)` — перезаписывает список, `doc.deleteIn(["replacements", key])` — удаляет дубли,
  - `noop` сигнализируется через изменяемый флаг `changed` в замыкании (см. §8 LESSONS_LEARNED).

### `app/src/components/EditPanel.tsx`

- Импорт расширен: `dedupByTo` из `../lib/yaml-edit`.
- Состояние `pending`/`pendingBefore`/`setErr` переиспользуется как для добавления записи — без новых state/stores.
- Новый хендлер `handleDedup`:
  ```ts
  const res = dedupByTo(activeEntry.raw);
  if (!res.ok) { setErr(...); return; }
  if (res.noop) { setErr("Дублей по to нет — словарь чист"); return; }
  setPendingBefore(activeEntry.raw);
  setPending(res.raw);
  ```
- В `edit-panel-header` рядом с «↶ Undo» добавлена кнопка «⚙ Дедуплицировать»:
  - видна только при `activeKind === "replacements"`,
  - `disabled` при `pending !== null` (если уже открыт другой preview).
- Список правил `edit-list` теперь показывает `key (to)` для `replacements`/`lemma_replacements`:
  ```tsx
  <span>
    {key}
    {to ? (<> (<code>{to}</code>)</>) : null}
  </span>
  ```
- Тип `existingKeys` изменён с `string[]` на `Array<string | { key: string; to?: string }>` — для `glossary`/`lemma_irregular` остаются строки (там `to` нет).

## Что НЕ сделано (вне MVP, осознанно)

- Дедупликация по `to` в `lemma_replacements` — другая семантика, отдельная фича.
- Дедупликация по «очень похожему `to`» (`Codex` vs `кодексе` — это разные слова, не дубль; определяется вручную).
- Автоматический пуск при «Применить» формы (когда пользователь пытается создать новое правило `to=Python` при существующем). Пользовательский запрос именно про ручную разовую кнопку.
- Mass-merge нескольких `to` в один через диалог с выбором, какое `to` оставить.
- Фонематическое/стемматическое сравнение.

## Lessons Learned (дубль §8 — для удобства поиска)

### 12. Операция, которая «ломает» словарь → пропускать через превью-diff

`dedupByTo` затрагивает множество правил: удаляет часть из них, переписывает `from` других. Это крупное изменение, которое пользователь должен увидеть до применения. Решение — переиспользовать существующий `pending`/`Превью`/`Применить`: пользователь видит diff, может откатить через Undo. Никаких новых кнопок не понадобилось.

### 13. `noop` через изменяемый флаг, не через сравнение строк

`applyEdit` возвращает `EditResult`, и в нём нет `noop` — только `raw`/`ok`/`error`. Хочется сказать вызывающему «не было изменений». Сравнение `res.raw === raw` ненадёжно: CST-операции могут переформатировать пробелы/переводы строк без семантических изменений, и строка получится «другой». Чистое решение — замыкание: `let changed = false; ...; changed = true; ...; return { ...res, noop: !changed }`.

### 14. Пустые `to` — пропускать, не мерджить

Правила без `to` (битые/незаполненные) при нормализации дают пустую строку и слиплись бы в одну группу «без имени». Защита — `if (!normTo) continue;` в обходе пар. Без этой защиты можно случайно слить десяток мусорных записей в одну.

## Сценарий ручной проверки

(для следующего, кто будет смотреть)

**Проверка списка правил с `(to)`:**

1. Открыть `sample/transcript_optimizer` → вкладка `replacements`.
2. В правой панели видим, например: `replacement_rule_015 (n8n)`, `replacement_rule_037 (Python)`, `replacement_rule_043 (Python)`, `replacement_rule_049 (плохо)`, `replacement_rule_050 (плохо)`.
3. Для `lemma_rule_001 (Slack)` тоже видно `(to)`.
4. Для `glossary` — только `id` без `(to)`.

**Проверка дедупликации:**

1. Нажать «⚙ Дедуплицировать» в правой панели.
2. В превью-diff видно:
   ```
   - replacement_rule_037:
   -   to: Python
   -   label: tech_term
   -   from:
   -   - питанящих
   - replacement_rule_043:
   -   to: Python
   -   label: tech_term
   -   from:
   -   - питоновском
   + replacement_rule_037:
   +   to: Python
   +   label: tech_term
   +   from:
   +   - питанящих
   +   - питоновском
   ```
3. «Применить» → в словаре осталось `_037 (Python)` с `from: [питанящих, питоновском]`, `_043` удалён.
4. Аналогично — для `плохо` (`_049`/`_050`).
5. Ещё раз нажать «⚙ Дедуплицировать» → сообщение «Дублей по to нет — словарь чист».
6. Откат через ↶ Undo возвращает исходное состояние (с дублями).
7. **Регистр:** создать вручную правило `_xxx: { to: python, from: [x] }` при существующем `_yyy: { to: Python, ... }`. «Дедуплицировать» → мёрдж по lowercase: `python` и `Python` — один и тот же нормализованный `to`.

**Что НЕ сломано:**

- Точные дубликаты `from` (как в фиче «Похожие from») — поведение не изменилось.
- Monaco markers `replacements:to-duplicate` — после применения дедупа исчезают (правильно), до применения — остаются.
- «Превью»/«Применить»/«Отмена» для добавления нового правила — работают как раньше.
