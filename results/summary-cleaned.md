### 1️⃣ Доклад Mikhail Shvedkov: процедурная память агентов

**Ключевые наблюдения (на примере DeckBuddy):**
- На доклад в ~10–15 тыс. токенов итоговый расход составил ~50 тыс. токенов: ~1/3 уходила на объявление Skills, MCP-тулов, системных промптов; ещё ~40% — на генерацию HTML-кода презентации.
- Исследования подтверждают: до 60 тыс. токенов может уходить только на описание возможностей модели.
- Opus на идентичном промпте потратил ~100 тыс. токенов и сломал саму структуру презентации; Composer'ы дают более предсказуемый результат (~$1.5 за 10 слайдов).

✅ **Принятые решения и рекомендации:**

- **Не давать агенту доступа** к инструментам, не нужным для решения текущей задачи.
- **Повторяющиеся задачи** (чаще одного раза) — оформлять как Skills.
- **Строго детерминированные последовательности действий** — выносить во внешний workflow/script/Tool; агенту оставлять только вызов.
- **Разделять творческую и детерминистическую части** задачи.
- При большом числе тулов модель начинает ошибаться в их выборе, теряет контекст и галлюцинирует — overhead по деньгам, скорости и качеству одновременно.
- Использовать **промежуточный DSL вместо чистого Markdown** в Skill — модель генерирует разметку, скрипт превращает её в HTML. Экономия: ~20 тыс. токенов на презентацию при минимальной потере качества (Markdown-токены выросли с 3 000 до 3 200, что пренебрежимо).
- Workflow полностью без LLM на генерации HTML даёт моментальную генерацию и нулевой расход токенов, но **полную потерю качества** — пригодно только там, где последствия некритичны.

📋 **Планы и Action Items:**

- Вынести генерацию HTML в отдельный скрипт + Skill с инструкцией вызова скрипта.
- Повторно обсудить выбор инструментов для детерминированных оркестраторов (LangChain, LangGraph, n8n, Dify, обычные Python-скрипты) — **обсуждается**.
- Продолжить изучение подходов Hermes и самообучающихся моделей, которые сами генерят себе скрипты/Skills для экономии токенов.
- Andrey Potapov попросил поделиться шаблоном Cursor для подсчёта токенов между чатами — Михаил согласился.

### 2️⃣ Доклад Andrey Mospanov: Data Sculptor

**Контекст:** дата-инженер, загрузка исторических данных (с 2023) в новые таблички датамарта. Получено ~1 000 Excel-файлов с типовыми проблемами.

**Возможности приложения (Python + GUI, развёртывание через installer):**
- Массовое переименование файлов по маске (важно, т.к. дата парсится из имени файла).
- Конвертация Excel → CSV с выбором кодировки и разделителя.
- Просмотр структуры файлов в папке наглядно (с прогресс-баром).
- Удаление лишних колонок по имени.
- Swap columns между собой.
- Добавление пустой колонки с заголовком в любое место (для совместимости с новой структурой даталида).
- Указание конкретного листа (Sheet 1) при чтении.
- Русифицированный интерфейс.

✅ **Принятые решения и рекомендации:**

- **Сначала наточить топор, потом рубить лес:** потратить 2 дня на инструмент, чтобы затем обработать батчами по 150–200 файлов.
- Оптимизация чтения: вместо полного чтения всех строк читать только заголовки — существенный прирост скорости.
- Использовать конкретные библиотеки Python (openpyxl/pandas) и GUI-оболочку вместо терминальной утилиты.
- При работе с датами в Excel учитывать формат «дни с 1900 года» — обработка реализована отдельно.

⚠️ **Рекомендация по безопасности (от Andrey Potapov):**
- Внутренние транскрипты и проектные данные **не отдавать** во внешние LLM (Cursor, Codex и т.п.) — есть риск утечки и неконтролируемого распространения.
- Использовать локальные решения (например, Minimax) для чувствительных данных.

- Потенциально: обернуть Data Sculptor как CLI-утилиту, чтобы её мог вызывать агент как Tool — **обсуждается**.
- Возможное продолжение темы оркестрации и безопасной работы с данными на следующей встрече (пятница) — **требует уточнения**.
